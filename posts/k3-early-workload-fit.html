K3 field notes

Early K3 workload-fit notes

The K3 board is already more practical than the RV2 for normal RISC-V development work. The early evidence also says something less marketable but more useful: it is a strong Linux development and vendor-INT8 computer-vision board, not a general-purpose LLM appliance.

Why this is interim

This is a snapshot from the first bring-up and workload-fit pass. Some tracks are strong enough to guide expectations now. Others still need external hardware, permission fixes, or cleaner runtime isolation before they deserve stronger claims.

The useful framing is workload fit, not headline hardware labels. K3 has normal X100 cores, vendor-mediated A100/TCM paths, media blocks, a PowerVR GPU stack, and visible high-speed I/O potential. Those pieces do not all behave like ordinary Linux CPU cores, and they do not automatically accelerate arbitrary models.

The short version

  • K3 is a much stronger RISC-V Linux development board than RV2.
  • Normal user sessions get eight X100 cores, not sixteen ordinary cores.
  • Vendor-prepared INT8 computer-vision models are the clearest AI win so far.
  • YOLO numbers should be reported as full pipeline timing, not just raw inference latency.
  • H.264/H.265 media acceleration is real, but normal-user device access still needs cleanup.
  • OpenCL/Vulkan support is real, but current compute evidence is narrow and kernel-shape-sensitive.
  • Small local LLMs run, but current evidence does not support calling K3 a general LLM board.
  • 10GbE and PCIe/NVMe potential should not be treated as measured throughput yet.

Normal Linux work looks good

The cleanest early result is ordinary development work on the X100 cores. A generated C++20 build with 64 translation units scaled well across the eight normal user-visible cores.

Build mode Wall time Read
make -j1 160.042 s Serial baseline.
make -j8 21.431 s About 7.47x faster.
Monitored thermal sample 67.0 C hottest zone No sampled X100 frequency drop in that run.

That makes K3 feel like a real local development board rather than just a target to cross-compile for. The RV2 can run useful lab tasks, but K3 is much easier to imagine using for builds, package work, and longer interactive sessions.

The CPU/RVV story changed

The matched RV2/Pi 400 primitive harness puts K3 in a much better place than RV2. K3 is broadly ahead of RV2 and has several wins against the old Pi 400 comparison, especially where explicit RVV and streaming memory matter.

Primitive K3 result Interpretation
Scalar u64 mix 1340.9 M ops/s Much stronger scalar baseline than RV2.
64 MiB pointer chase 6.39 M steps/s Roughly tied with the old Pi 400 result and much better than RV2.
memcpy 5.72 GB/s Strong streaming-memory signal.
Explicit RVV int8 dot 7.61-9.84 GMAC/s RVV is useful when directly targeted.
RVV 1x4 tile up to 10.87 GMAC/s Output reuse and tile shape matter.

The lesson from RV2 still applies: compiler auto-vectorization is not the whole story. Explicit RVV kernels and shape-aware tiling are where the board starts to look interesting.

The clearest AI fit is INT8 vision

The strongest accelerator result so far is not chat-model inference. It is vendor-prepared INT8 computer vision through the SpaceMIT ONNX Runtime path. Official INT8 CV models show large raw provider wins over CPU.

Model CPU SpaceMIT Speedup
ArcFace INT8 140.140 ms 1.882 ms 74.45x
MobileNetv2 INT8 77.422 ms 3.216 ms 24.08x
ResNet50 INT8 299.356 ms 5.149 ms 58.14x
YOLOv8n 192x320 INT8 59.717 ms 2.262 ms 26.40x
YOLOv8n 320x320 INT8 83.521 ms 2.911 ms 28.69x

This is the best current positive AI claim: vendor-prepared INT8 CV models can be very fast through the SpaceMIT provider. It should not be generalized to arbitrary ONNX graphs. Some synthetic attention-like, reduction, elementwise, and small quantized graphs were weak or slower through the provider than CPU.

YOLO pipeline timing matters

The YOLO wrapper evidence is useful because it includes more than raw model execution. Python postprocessing and OpenCV read/decode costs are already large enough to affect application-level FPS.

YOLO video probe Mean total Notes
192x320, no draw 19.898 ms/frame Real-object H.264 clip with stable detections.
192x320, draw 22.350 ms/frame Drawing adds visible cost.
320x320, no draw 23.936 ms/frame Higher input size still viable.
320x320, draw 27.491 ms/frame Pipeline cost, not raw accelerator latency.

That is still a promising edge-vision shape. The guardrail is that application timing should include decode/read, preprocessing, inference, NMS/postprocess, and drawing when drawing is part of the application.

Media is real, but permissions are not boring yet

FFmpeg/stcodec media acceleration is another strong lane. In the 30-second 1080p30 sweep, the hardware paths were comfortably faster than real time.

Media path Result
H.264 stcodec encode about 5.39x real-time
HEVC stcodec encode about 8.29x real-time
HEVC stcodec decode about 10.1x real-time

The catch is operational. Normal-user access to the needed video device path is not clean yet. Until that is fixed and rerun from a fresh normal session, this is a real media capability but not yet an ergonomic user story.

A100 is not eight extra shell cores

Linux exposes more CPU topology than a normal user process can simply use. The observed user session gets the X100 side. A100 is reached through vendor-aware runtime paths, not by running arbitrary shell work on CPUs 8-15.

The current practical model is:

X100 Linux process
  -> vendor-aware runtime
  -> A100/TCM placement and DMA mappings
  -> supported dense tensor primitive
  -> result returned to the application

That distinction matters. SpaceMIT ONNX Runtime can map the provider libraries and A100/TCM device paths for supported workloads. Direct ordinary X100-side TCM byte-loop reads were slow in the first probe. Treat A100/TCM as a vendor-runtime resource, not as a general scratchpad or extra pool of Linux cores.

GPU/OpenCL is real, but narrow so far

The PowerVR OpenCL and Vulkan stacks enumerate and run smoke tests. A controlled non-vendor OpenCL FP32 kernel validated correctly, and explicit local sizing improved one simple recurrence kernel to about 38 GFLOP/s by wall-clock timing.

That is enough to say the GPU runtime is real and worth experimenting with. It is not enough to say the GPU is broadly fast for compute. A simple hand-written OpenCL SGEMM topped out around 1.67 GFLOP/s at N=512, while NumPy/BLAS CPU reached about 3.06 GFLOP/s on the same size. Event profiling was not trustworthy in the controlled harnesses, so validated wall-clock timing is the standard for now.

Local LLMs run, but the big claim is not ready

The board can run small local GGUF models. Qwen3 0.6B Q4_0 ran at 22.574 prompt tok/s and 15.812 generate tok/s at 8 threads. LiquidAI LFM2.5 8B A1B Q4_0 also fit and ran locally at 9.637 prompt tok/s and 7.828 generate tok/s.

That is useful, but it is not the same as proving a broad LLM appliance. The clean small-model control that compared paths showed the generic X100 llama.cpp route beating the SpacemiT llama path on SmolLM2 135M Q4_K_M. Later thread-sweep results showed good scaling for the installed backend, but also showed why backend isolation matters: different frontends can end up loading the same installed backend libraries.

The IME2/low-bit tensor work is promising and deserves its own later write-up. Manual Q4_0 tensor paths have reached real Qwen tensor bytes through hidden SpacemiT placement and validated against references. The missing public claim is stock model-level placement: ordinary model runs should not be credited with IME2 acceleration until attribution proves those tensors actually use the hidden path.

What should wait

Several K3 tracks are interesting but not publishable as strong claims yet:

  • 10GbE: the capable interface is visible, but no-carrier in the current tests and no measured 10GbE peer throughput yet.
  • Wired routing/firewall throughput: current TCP smoke results used the reachable Wi-Fi route.
  • NVMe/PCIe storage: no NVMe endpoint was present in the current inventory.
  • Camera or stcodec-to-detector pipeline: media permissions need to be fixed before the clean normal-user path is proven.
  • Large GGUF/30B-class claims: need deliberate model selection, checksums, load-time/memory measurements, backend isolation, and attribution.
  • Broad GPU compute: needs tuned GEMM, convolution/image kernels, Vulkan compute controls, and repeated CPU comparisons.

Current recommendation

If someone is evaluating the K3 today, I would steer them this way: use it as a capable RISC-V Linux development board first, use SpaceMIT ONNX Runtime for vendor-prepared INT8 computer vision, treat media acceleration as real but still needing permission cleanup, and treat A100/TCM as a runtime-mediated accelerator path.

I would not buy it primarily as a general LLM server from the evidence so far. It can run local models, and the hidden low-bit acceleration path is genuinely interesting, but the practical public advice is still conservative: measure the exact model, quantization, backend, dispatch path, and pipeline timing before believing the label on the box.
