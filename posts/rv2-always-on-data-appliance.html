Appliance notes

The Orange Pi RV2 makes a good always-on data appliance

The Orange Pi RV2 is not a faster Raspberry Pi for ordinary Linux work. Its more interesting niche is quieter: keep many public data feeds open, reduce them into compact aggregate state, and leave enough CPU and memory headroom for the machine to run unattended.

The question

After the earlier stream-appliance prototype survived 12 hours of Nostr and crypto aggregation, I wanted a more realistic mixed workload. Could the board run multiple live WebSocket feeds, market order-book aggregation, and periodic public-data polling at the same time without turning into a one-core bottleneck?

The six-hour soak answered that clearly enough to change how I think about the board. This is not a general-purpose server story. It is an edge reducer story.

What ran

The soak combined three branches that would normally be tested separately:

  • A Rust AISStream client receiving live global ship-position-style messages and keeping aggregate counters.
  • The Python stream appliance processing Nostr, Bluesky Jetstream, Coinbase/Kraken trades, and Coinbase/Kraken order books into SQLite feature buckets.
  • A periodic open-government metadata sweep polling NWS, USGS, FIRMS, NEXRAD, and GOES-style sources every five minutes.

The job deliberately did not archive raw social posts, raw AIS messages, raw trade messages, radar products, or satellite products. It kept compact aggregate rows, counters, feature buckets, and source summaries.

The six-hour result

The headline result is simple: all of that ran together for six hours while the steady live workload stayed well below one core.

Branch Work done CPU Memory
AISStream Rust client 6,327,573 messages, 3.50 GB, 292.9 messages/s 4.0% median 10.2 MB median RSS
Main stream appliance Nostr, Jetstream, trades, order books, SQLite buckets 15.6% median 65.4 MB median RSS
Government polling 72 metadata polls at five-minute intervals brief spikes short-lived process

The AIS process peaked at 4.7% CPU. The main appliance peaked at 17.6% sampled CPU. The two long-lived live processes together stayed under 100 MB of memory. The aggregate SQLite database was about 8.2 MB at the end.

AIS was the surprise

AISStream turned out to be a useful live-data source for this board. It is much larger than the Nostr relay set I had been using, but still cheap after moving the client to Rust.

The operational lesson was as important as the performance number: public WebSockets reset. The first longer AIS run exposed that immediately. Once the Rust client reconnected and resumed until the target duration expired, the source became a realistic always-on branch rather than a fragile demo.

The aggregate database stayed small

The main stream appliance kept one-minute buckets and aggregate-only feature tables. The result is the kind of database size that can be copied, backed up, queried, and inspected without pretending the board is a data lake.

This is the part that makes the appliance shape feel practical. The RV2 is not winning by storing everything. It is winning by refusing to store everything.

Government data splits into two classes

The public-data sweep was included to test another kind of workload: authoritative feeds and object listings rather than live social or market streams. It split cleanly into signal feeds and bulk object feeds.

Source Observed shape Appliance role
NWS active alerts hundreds of active alerts low-volume authoritative event feed
USGS earthquakes around 200 all-day events low-volume trigger feed
NASA FIRMS VIIRS tens of thousands of rows good geospatial event feed
NEXRAD Level II listing first page around 5.9-6.2 GB listed safe to index, expensive to download
GOES current-hour listing roughly 310-366 MB listed regular heavy object stream

Metadata polling was safe. Bulk radar and satellite products are different. A board like this can watch those feeds, index them, choose a subset, and enforce byte budgets. It should not blindly download every interesting object just because the listing is easy to fetch.

The capacity lesson

The RV2 still has obvious weaknesses. It is not a magic general CPU, and Python hot loops hit limits quickly when the event rate rises into the thousands per second. But this workload is not trying to be a high-QPS service. It is an always-on reducer.

The practical model now looks like this:

  • Use Python for orchestration, dashboards, experiments, and low-rate polling.
  • Use Rust for high-volume live clients and reducers.
  • Store aggregate rows, not raw firehose archives.
  • Treat NEXRAD, GOES, and similar product feeds as byte-budgeted object streams.
  • Assume public WebSockets will reset, and make reconnect behavior part of the design.

Where this leaves the RV2

This is the clearest practical role I have found for the Orange Pi RV2 so far. It is not a Raspberry Pi replacement for every task. It is a small RISC-V box that can sit on a network, keep many useful public feeds warm, reduce them into privacy-preserving aggregate state, and expose a compact dashboard or database for later analysis.

That does not make it fast in the abstract. It makes it useful in a specific way: continuous ingest, cheap feature extraction, compact persistence, and enough headroom to stay boring.

No analytics, comments, or tracking scripts are used here.
