opencode now has a RISC-V single-file package

The opencode work has moved past the earlier experimental package and now has a second stable forward port. The current 1.14.48 Linux RISC-V tarball has installer support, shell completions, package-local smoke tests, an RV2-validated upstream TUI attach path, and a separate patch bundle that shows the port can move with upstream.

Download

The package is published as opencode-1.14.48-linux-riscv64-singlefile.tar.gz. It contains the RISC-V opencode executable, runtime sidecars, native PTY support, the RISC-V Parcel watcher package, shell completions, release notes, build metadata, checksums, and bundled license notices.

This is a patched RISC-V package, not an official upstream opencode release. The bundled BUILDINFO records dirty source trees for opencode and Bun, so treat the artifact as a practical RV2 build rather than a byte-for-byte rebuild recipe.

Why 1.14.48 matters

The useful result is not just a newer tarball. The 1.14.48 package proves the RISC-V work is now a forward-portable patch set instead of a one-off local tree. The release includes opencode-1.14.48-riscv-forward-port.tar.gz, which contains a small entry overlay and a tracked source patch that can be applied to a clean upstream checkout at 4d8ac17c263c0e58fb99c25cc636684dffb45a50.

tar -C /path/to/opencode -xzf 0001-riscv-entry-overlay.tar.gz
git -C /path/to/opencode apply 0002-v1.14.48-riscv-source-compat.patch

That split is the durable part. The overlay carries the RISC-V command entrypoint and diagnostic modules, while the patch carries the compatibility changes: the filesystem export needed by the package path, TUI smoke hooks, clean exit tracing, guarded renderer teardown, plugin runtime tracing, and a smoke-only external plugin skip. When upstream moves, those are the pieces to rebase and re-run through the package smoke.

What it took

This is the part that would have saved the most time at the start: the opencode port was not blocked by one missing target string. The work had to cut through several layers of runtime and packaging assumptions.

  • Bun had to accept and produce a RISC-V standalone executable target for this package path.
  • The package needed a RISC-V Bun runtime with JavaScriptCore, ICU, libc++, libc++abi, and libunwind sidecars available at runtime.
  • The opencode entrypoint was built through a Bun single-file compile path instead of the upstream release matrix, which currently centers common x64 and arm64 targets.
  • rust-pty had to be available as a RISC-V native library so terminal/session paths could load and spawn correctly.
  • @parcel/watcher needed a Linux RISC-V glibc native package so file watching used the native inotify backend instead of silently falling back.
  • The MCP stdio path needed a RISC-V standalone-safe entrypoint shim to avoid a current Bun standalone import failure around direct node:process use.
  • The plugin path needed to move from a standalone loader mode to the upstream plugin service with SDK-backed auth hooks and event forwarding.
  • The TUI path had to move past a focused OpenTUI/Solid attach shim to the upstream TUI app graph through a RISC-V attach wrapper.
  • OpenTUI's native renderer library had to be extracted to a real filesystem path before dlopen, because shared libraries cannot be opened directly from Bun's embedded filesystem.
  • The Linux clipboard fallback had to avoid the current Bun RISC-V standalone clipboardy import crash by using native clipboard commands and terminal OSC 52.
  • The release package needed install/uninstall scripts, completions, package-local smoke, release notes, build info, and checksum coverage for those files.

What passed on RV2

The latest package smoke is broader than the previous experimental check. It verifies the tarball checksum, extracts the package, verifies package checksums, starts the server, exercises the upstream TUI attach path under a PTY, runs the package-local smoke, installs through install.sh, verifies the installed wrapper, runs opencode --riscv-smoke, then uninstalls and checks cleanup.

Install

The package can be run in place or installed under a prefix. The installer writes a wrapper that sets LD_LIBRARY_PATH for the bundled runtime sidecars and installs bash, zsh, and fish completions.

sha256sum -c opencode-1.14.48-linux-riscv64-singlefile.tar.gz.sha256
tar -xzf opencode-1.14.48-linux-riscv64-singlefile.tar.gz
cd opencode-1.14.48-linux-riscv64-singlefile
sha256sum -c SHA256SUMS
./smoke.sh
./install.sh --prefix "$HOME/.local"
opencode --version
opencode --riscv-smoke

The installer also supports uninstalling the wrapper, completions, and package root:

./install.sh --prefix "$HOME/.local" --uninstall

The package metadata records the executable, runtime libraries, librust_pty.so, and the RISC-V Parcel watcher native module. License coverage is packaged under LICENSES/, with a short THIRD_PARTY_NOTICES.md describing the bundled components.

What changed since the experimental build

The first published opencode artifact proved the single-file path could run on RV2. This newer package removes several of the most important qualifications.

  • The package name no longer carries the experimental suffix.
  • install.sh, smoke.sh, shell completions, BUILDINFO, and RELEASE_NOTES.md are included in the tarball and covered by SHA256SUMS.
  • The TUI path now imports the upstream TUI app graph through a RISC-V attach wrapper instead of the earlier focused attach shim.
  • The package smoke now verifies internal TUI plugin activation, prompt readiness, tui:done, renderer teardown, and PTY status 0.
  • The plugin runtime now uses the upstream plugin service by default, including real SDK-backed GitLab and Poe auth hooks.
  • The MCP path uses the upstream MCP service for local stdio servers and tool discovery.
  • The installer and installed wrapper have their own smoke path, so users can verify the installed package without remembering the runtime-library setup.

What changed since 1.14.25

The main change is process quality. The package shape stayed stable while the port moved forward to a newer upstream opencode ref. The shipped tarball was rebuilt, checksummed, and smoked as a release artifact, and the RISC-V-specific changes were preserved in a public forward-port bundle rather than being left as only a local working tree.

The final package smoke still covers the same release-critical path: tarball checksum verification, package checksums, metadata and license files, server startup, upstream TUI attach readiness, clean PTY exit, package-local smoke, install, installed wrapper smoke, and uninstall.

What still matters

The package still carries the realities of a RISC-V port. It bundles runtime sidecars for the validated RV2/K1 class environment, and the OpenTUI native library is still extracted to a real temporary filesystem path before renderer startup because shared libraries cannot be opened directly from Bun's embedded filesystem.

The upstream clipboardy fallback is replaced in this single-file mode with Linux-native clipboard commands and terminal OSC 52 because the current Bun RISC-V standalone path still crashes on that import. That is a scoped workaround, not a user-facing showstopper for the validated smoke path.

The practical lesson is that "RISC-V support" became real only after the runtime, native modules, plugin service, MCP service, TUI renderer, installer, and smoke tests were each made explicit. The useful artifact is the tarball, but the useful knowledge is the boundary list.
