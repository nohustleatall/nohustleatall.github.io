K3 LLM notes

The K3 LLM speed limit is not one number

The most useful K3 LLM question is no longer "is the 60 TOPS claim real?" A raw sparse-int4 path makes the headline plausible. The better question is how much of that peak can survive contact with real model shapes, quant formats, attention, routing, logits, and runtime scheduling.

The current answer

The K3 looks much more promising for language-model experiments than it did at first, but the speed limit is architectural. Raw IME throughput, synthetic projection throughput, synthetic decode skeletons, and off-the-shelf model throughput all measure different layers of the stack.

So far, LFM2.5 8B A1B is the most important clue. It is not the largest public model tested, and it is not the fastest tiny control. It is interesting because its shape lines up unusually well with what the board wants: hidden size 2048, only 6 attention layers out of 24, many shortconv/mixer layers, top-4 MoE routing, and packed Q4_0 expert tensors.

Four ceilings

I now think about K3 LLM performance as four ceilings, from least realistic to most realistic:

Layer What it says What it does not say
Raw IME peak Native sparse-int4 instructions can reach tens-of-TOPS class throughput. That a GGUF model will generate tokens at anything close to that rate.
Ideal tensor shapes Q4_0 and some K-quant shapes can reach hundreds of GOPS when the tile is filled. That batch-one decode will look like prompt processing.
Synthetic runtime skeletons Windowed or hybrid attention plus persistent projection workers can move the theoretical decode budget. That a trained model has the same dataflow, quality, or graph overhead.
Real public models Current LFM2.5 and Gemma/Qwen runs show what survives in llama.cpp today. That the hardware has no more headroom.

The raw peak is real, but narrow

The raw IME sparse-int4 probe measured about 47.2 sparse-effective TOPS with eight A100-side workers. Scaling the observed clock toward the advertised max puts that in roughly the low-50s sparse-effective TOPS range. That makes the K3's headline class plausible.

But this is a direct primitive result. It does not include model loading, tensor layout conversion, repacking, routing, attention, residual work, logits, sampling, correctness validation for a full model, or stock llama.cpp dispatch. It explains the silicon class. It does not give a token rate.

Shape is the first speed limit

The ideal-shape benchmarks explain why prompt processing often looks much better than generation. The native tile wants width. Single-token decode has very little reuse, while prompt or batched work can reuse the same weights across many columns.

This is why the practical K3 advice keeps coming back to model and quant layout. Q4_0 is the cleanest fast path in the probes. Q4_K is more common in useful public artifacts. Q5_1 and Q5_K can be acceptable. Some attractive formats elsewhere are bad first targets here if they miss the native path.

A naive LFM synthetic test was misleading

The first LFM-shaped synthetic result looked disappointing. It modeled the active MoE experts as separate sparse projections, and the top-4 LFM-like case landed around 3.8 to 4.2 tok/s depending on the window setting. A top-1 ablation reached about 6.5 tok/s, which mostly proved that the projection path was being modeled as too expensive.

The problem was the abstraction. Real LFM-style MoE in llama.cpp does not have to behave like a handful of independent projection calls. The useful path is packed and expert-id-aware.

The packed expert path changes the ceiling

A later synthetic test matched the packed MUL_MAT_ID expert operation more closely: Q4_0 expert tensors, F32 activations, I32 expert ids, 32 experts, and top-4 routing. That result moved the expert projection path out of the obvious-bottleneck category.

Packed LFM-like expert op Tokens Median time Throughput
gate/up 1 0.353 ms 83.08 GOPS
down 1 0.362 ms 81.17 GOPS
gate/up 128 11.854 ms 317.04 GOPS
down 128 11.762 ms 319.52 GOPS

For a rough expert-only decode budget, gate plus up plus down costs about 1.068 ms per MoE layer in this synthetic setup. Across 22 MoE layers, that is about 23.5 ms/token, or roughly 42.6 tok/s before the rest of the model is counted. That is not a full model result, but it is an important shift: the packed expert path is probably not the sole reason real LFM2.5 decode is around the high teens today.

Why LFM2.5 is close to the board's preferred shape

The LFM2.5 8B A1B anatomy is unusually convenient for this board:

  • hidden size 2048;
  • 24 total layers;
  • only 6 full attention layers;
  • 18 shortconv/mixer layers;
  • 32 experts with 4 active per token;
  • MoE intermediate width 1792;
  • about 91.5% of parameters in expert tensors;
  • public Q4_0 expert tensors that can be routed through the packed expert path.

That is close to the shape the synthetic work keeps pointing toward: modest hidden size, limited attention frequency, packed MoE, and quant formats the backend can accelerate. It is a very different target from "any 8B model" or "any 30B MoE model."

The current bottleneck hypothesis

If the packed expert synthetic result is directionally right, then LFM2.5's remaining gap is probably not just "the experts are slow." The likely remaining costs are the dense first layers, shortconv/mixer blocks, attention layers, router/top-k, norms and elementwise work, graph scheduling, memory movement, and the tied embedding/output projection.

The output projection already showed up once. The original public LFM2.5 Q4_0 GGUF had a lone Q6_K tensor for token_embd.weight, apparently tied to the vocabulary projection. In the opt-in path, perf still sampled the generic Q6_K dot kernel heavily. A derived model with that tensor requantized to Q4_0 moved decode from about 15.3 tok/s to about 18.0 tok/s. That is a speed lever, but it changes logits and needs task-level validation before being treated as a default artifact.

Attention still decides the runtime class

The native attention sweeps explain why LFM-like sparse attention schedules matter. Plain attention score/value cost grows linearly with context. Synthetic h2048 attention grew from about 6.6 ms/layer at context 1024 to about 210 ms/layer at context 32768. Full attention in every layer is a bad fit for fast decode on this board.

The more encouraging synthetic track is hybrid or windowed attention. With vectorized fast-math attention, local-window h2048/window512 skeletons land around 10 tok/s when every layer pays the attention tax. A hybrid attention-every-four-layers schedule gets into the mid-20s tok/s synthetic range. That is why LFM2.5's 6 attention layers out of 24 matter so much.

What this means in practice

The practical K3 target is not "any model that fits in memory." It is a narrower class:

  • hidden size around 2048 or 2560;
  • few full attention layers, local attention, or hybrid attention;
  • packed MoE expert operations rather than many separate projection calls;
  • Q4_0, Q4_K, Q5_1, Q5_K, or Q8_0 hot tensors in backend-friendly shapes;
  • minimal hot-path repacking;
  • a vocab/output projection that does not fall back to a slow generic dot path.

That is the current speed-limit story. The hardware has real headroom, and the raw TOPS result is not fake. But useful tokens are gated by architecture and runtime integration. LFM2.5 looks close to the best off-the-shelf shape found so far because it avoids several of the traps at once.

Next useful measurements

The next experiment should be stage-level timing for LFM2.5: expert MoE time, shortconv/mixer time, attention time, router/top-k time, norms and elementwise time, and output projection time. That would turn the current hypothesis into a budget and make it clear whether the next gains come from kernels, graph scheduling, quant layout, or model preparation.
