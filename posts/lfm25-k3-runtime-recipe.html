K3 runtime notes

LFM2.5 on K3: the runtime recipe matters as much as the model

The latest LFM2.5 work changed the practical K3 story. The model shape was already good. The new lesson is that the real speed unlock is keeping the model on the K3-friendly repacked Q4_0 path, then using different thread settings for decode and prefill.

The short version

The best current runtime policy is simple:

decode threads:        6
batch/prefill threads: 8
repack:                enabled
format:                Q4_0 fast-path friendly

With that policy, the base LFM2.5 Q4_0 model is now a real low-20s tok/s decode target on K3, not just a synthetic shape argument. A risky derived model with the token embedding/output-side tensor requantized to Q4_0 reaches about 25 tok/s at short context and about 22.7 tok/s after a 1602-token prefill. That derivative is not promotion-ready because quality has not been validated.

The practical speed result

The current measured path puts LFM2.5 in a different class from dense Gemma-style runs. The packaged SpacemiT llama-bench path measured 122.857 tok/s prompt processing at pp128 and 22.010 tok/s decode at tg32. The target-only harness then pushed realistic one-shot prefill and decode runs further.

Runtime setting 1602-token prefill Decode after prefill
--threads 6 --threads-batch 6 70.404 tok/s 20.645 tok/s
--threads 6 --threads-batch 8 88.725 tok/s 20.686 tok/s
--threads 8 --threads-batch 8 86.740 tok/s 20.496 tok/s

The split setting matters because decode and prefill are different workloads. Decode likes 6 threads. Real batched prefill still wants 8 batch threads. Treating them as one knob leaves performance on the table.

Context penalty is real, but gradual

One useful result is what did not happen: LFM2.5 did not collapse immediately as prompt length grew. With the 6/8 thread split, decode dropped about 8.3% from a 102-token prompt to a 1602-token prompt.

Prompt tokens Prefill Decode after prefill
102 94.906 tok/s 22.500 tok/s
202 94.936 tok/s 22.373 tok/s
402 92.887 tok/s 22.047 tok/s
802 92.578 tok/s 21.611 tok/s
1602 87.528 tok/s 20.623 tok/s

That supports the earlier architecture read: LFM2.5 is not paying full-attention costs in every layer. Context still costs something, but over this range it is not the dominant limiter.

The Q4_0 token-embedding derivative helps

The public LFM2.5 Q4_0 GGUF has a token embedding/output-side tensor that keeps a slower path visible. A derived model with that tensor requantized to Q4_0 consistently improved decode by about 10%.

Model Prompt tokens Prefill Decode
Base Q4_0 102 94.841 tok/s 22.534 tok/s
Token embedding Q4_0 derivative 102 98.160 tok/s 25.009 tok/s
Base Q4_0 1602 88.675 tok/s 20.629 tok/s
Token embedding Q4_0 derivative 1602 90.361 tok/s 22.705 tok/s

A five-prompt smoke test showed the speed gain is stable: every prompt landed around 1.10x faster on decode. It did not show gross corruption. It also did not clear quality. Both base and derivative outputs exposed reasoning-style text in the harness, and the derivative had some prompt-shape drift. The right conclusion is "real runtime lever, not a recommended artifact."

MXFP4_MOE was a trap

The sprint also tested an MXFP4_MOE derivative. It ran, but it was much slower: about 1.918 tok/s decode versus 22.733 tok/s for the base LFM2.5 Q4_0 run in that comparison.

That is a useful negative result. A nominally lower-bit format is not automatically better on K3. What matters is whether the backend has a real accelerated path for that exact layout. This conversion moved too much work away from the fast Q4_0 path.

Where the time goes now

The steady decode profile is not dominated by sampling, shell overhead, or high-level application logic. It is mostly packed Q4 inner loops, memory copy/data movement around those kernels, IME2 GEMM kernels, OpenMP worker overhead, and a smaller remaining Q6_K-style path.

Delayed decode profile bucket Sample share
INNER_BLK_LOOP756 41.97%
memcpy_main_loop34 18.13%
spacemit_kernels::ime2::gemm_kernel_i8i8_m1 7.15%
OpenMP worker/wait symbols among top entries about 12.4%
Q4_0 repack in delayed profile 3.26%

greedy sampling 0.28%

The graph profile says the same thing at a higher level. The token-embedding Q4_0 derivative barely changes the FFN or MUL_MAT_ID bucket. It reduces output and generic MUL_MAT overhead. The main ceiling is still the packed expert path and the data movement around it.

The model-shape lesson survived

This sprint reinforces the earlier model-shape conclusion. LFM2.5 is fast on K3 because it has the right structure for this board: hidden size 2048, 24 layers, only 6 attention layers, packed Q4_0 MoE expert tensors, and a runtime path that can use MUL_MAT_ID instead of treating experts as separate projection calls.

A direct shape comparison against Gemma 4 E4B showed why this matters. The intrusive graph-profile numbers are not absolute speed numbers, but the composition is useful: LFM has less attention projection work, fewer layers, a smaller active graph, and a packed routed expert abstraction. Dense or generic Q4 FFN matmuls dominate Gemma. Packed routed expert matmul is the abstraction that makes LFM viable.

What to do next

The practical recipe today is:

  • use a Q4_0 fast-path-friendly LFM2.5 artifact;
  • keep repack enabled;
  • use 6 decode threads and 8 batch threads;
  • treat the token-embedding Q4_0 derivative as a speed experiment until quality is evaluated;
  • avoid exotic low-bit formats unless the backend has a measured fast path for the exact layout.

The next useful engineering target is not another broad model sweep. It is reducing copy/packing overhead around the packed Q4 expert path, or changing the expert MUL_MAT_ID path so it consumes a more directly usable activation and weight layout. The next useful model-quality target is a small deterministic eval or perplexity harness for the token-embedding Q4_0 derivative.

No analytics, comments, or tracking scripts are used here.
