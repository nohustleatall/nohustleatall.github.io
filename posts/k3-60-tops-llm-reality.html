K3 field notes

The K3's 60 TOPS claim is plausible. That still does not make it a 60 TOPS LLM box.

The K3 datasheet's AI headline is no longer just a mysterious marketing number. A raw sparse-int4 IME probe on the A100 side gets close enough to explain the class of the claim. The catch is that raw sparse-effective instruction throughput is very different from full local LLM token throughput.

The useful distinction

There are at least three different performance questions hiding behind one headline number:

  • Can the silicon execute native sparse int4 matrix instructions at a tens-of-TOPS class peak?
  • Can a runtime feed those instructions with the right sparse weights, masks, activations, and tile shapes?
  • Can a real language model generate useful tokens at a target speed and quality level?

The first answer now looks like yes. The second answer is workload- and runtime-dependent. The third answer still needs model-level proof for each model family, quantization, context length, backend, and dispatch path.

What was measured

The raw IME peak probe did not use GGUF, ONNX, llama.cpp, a model graph, or a quality test. It used direct inline assembly for public IME dot-product instructions and counted the work represented by the native tile shapes.

The toolchain needed the vendor extension enabled:

gcc -O3 -march=rv64gcv_xsmtvdotii -mabi=lp64d

That detail matters. Without the extension, the assembler rejects the instruction. Ordinary shell work on the K3 runs on the X100 side, CPUs 0-7. The raw IME instructions ran only when launched in a systemd scope restricted to CPUs 8-15, matching the earlier evidence that A100-side work is not just ordinary userland CPU work.

systemd-run --scope -p AllowedCPUs=8-15 ...

The raw peak result

With eight worker processes on the A100-side CPU set, the sparse int4 case measured about 47.16 sparse-effective TOPS.

Case Workers Dense TOPS Sparse-effective TOPS
dense i8 8 13.10 13.10
dense i4 8 26.10 26.10
sparse i8 8 11.75 23.51
sparse i4 8 23.58 47.16

The sparse-effective number is the relevant one for the headline class because 4:2 structured sparsity counts the skipped positions as represented dense work. That is normal for sparse accelerator marketing, but it has to be stated plainly.

The A100-side cores were observed around 1.8 GHz in this session, while the CPU table reports a 2.0 GHz max for CPUs 8-15. Scaling 47.16 TOPS by 2.0 / 1.8 gives about 52.4 sparse-effective TOPS. The rest of the distance to a 60 TOPS class peak is credible for a datasheet number: ideal scheduling, peak clock, unrolled vendor kernels, less loop overhead, and a cleaner instruction mix than a small userspace probe.

What the number does not mean

It does not mean a GGUF model sees 47 TOPS. It does not mean llama.cpp is using the path. It does not mean a 30B model has been locally reproduced above 10 tokens per second. It does not mean TCM residency has been proven for model tensors. It does not mean FP4 sparse model-tensor runtime support has been proven.

The measured raw path is narrow:

  • direct IME instructions;
  • A100-side CPU placement;
  • native sparse int4 tile shape;
  • no model loader;
  • no runtime repack cost;
  • no attention, routing, logits, or sampling;
  • no accuracy or output-quality claim.

Why model throughput is lower

Language-model runtime is not one ideal sparse dot-product loop. Even if a projection kernel is fast, the full token path includes model loading, tensor placement, quantization format conversion, repacking, activation handling, scheduling, attention and KV-cache work, residual paths, logits, and sampling.

The roofline estimate makes the decode problem concrete. Single-token decode is the danger zone because it combines low arithmetic intensity with underfilled native tiles.

Qwen-like down projection Cols GOPS Ops/byte estimate Lower-bound GB/s
q4_0 1 69.4 3.50 20.0
q4_0 8 360.8 24.98 14.6
q4_0 64 764.0 107.79 7.1
q4_K 1 63.3 3.31 19.2
q4_K 64 508.7 105.03 4.8

At one output column, the runtime streams a lot of weight data for very little reuse. At eight or more columns, the native tile starts to fill better. At 32 or 64 columns, the same weights are reused across many output columns, so arithmetic intensity improves sharply. That is why batched prefill and batched decode can look much better than single-stream token generation.

The model-shape lesson

The peak path wants a model and runtime shaped around it:

  • 4:2 structured sparse weights;
  • native int4 layout compatible with vmadot.sp;
  • recovery-mask and value streams packed the way the instruction expects;
  • batched work that fills the native N=8 tile;
  • minimal hot-path repacking;
  • attention that does not dominate decode.

Generic Q4_K or Q4_0 GGUF files are not automatically that. They may still run, and some current upstream SpacemiT paths are useful, but the raw sparse peak path is a different target from coercing ordinary GGUF tensors into a vendor backend.

Sparse model evidence is promising, but scoped

The sparse Llama work shows that real 2:4 sparse model weights can be decoded into something K3's sparse-native path can understand. A RedHatAI Sparse Llama 3.1 8B artifact uses 2:4 sparsity and int4 weight quantization. Its Marlin metadata decodes cleanly, and the recovered 2:4 symbols map to SpacemiT recovery masks.

The important narrow result: a real layer-0 self_attn.k_proj sparse weight tile was converted into the expected sparse values and masks, and the raw vmadot.sp i4 tile matched the scalar reference after the mask-packing convention was fixed. That is kernel/projection evidence. It is not full sparse Llama inference, not token throughput, not stock placement, not TCM residency, and not model-quality proof.

Runtime is the product

The later decode-skeleton work shows why a fast projection primitive is not enough. At short context, projection scheduling overhead matters. At medium and long context, attention score/value work can dominate.

Persistent projection workers cut a large part of the integration cost:

Shape Fresh workers Persistent workers
h2048, seven projections about 6.7-6.9 ms/layer 3.221 ms/layer
h2560, seven projections about 7.3-7.6 ms/layer 4.756 ms/layer

Attention is the other big lever. Full-context attention grows linearly with context. Local/windowed attention changes the budget class.

Shape Attention mode Estimated tok/s
h2048, 24 layers full 4k 1.38
h2048, 24 layers window 512 6.06
h2560, 24 layers full 4k 1.09
h2560, 24 layers window 512 4.52

Those are measured-component estimates, not a trained model run. They are still useful because they show the shape of a viable K3-native runtime: persistent projection workers, local/windowed attention, vectorized attention kernels, optional sink tokens, and careful fusion of q/k/v and gate/up work.

What to believe today

The conservative read is:

  • The K3's sparse-int4 peak claim is plausible as a sparse-effective raw instruction peak.
  • A100-side IME instructions are real and require the A100-side CPU set.
  • Raw peak measurements do not transfer directly to GGUF, llama.cpp, or chat throughput.
  • Single-token decode is memory/repack and tile-utilization limited.
  • Batched work, native sparse formats, and local/hybrid attention are the path toward useful LLM runtime performance.
  • No local 30B/A3B vendor-performance claim should be made without artifact, checksum, runtime, dispatch, attribution, context, and token-count evidence.

What I would test next

The next useful experiment is not another broad benchmark. It is a small runtime skeleton that combines the pieces already measured separately:

  • persistent projection worker pool;
  • real sparse projection calls;
  • fused q/k/v and gate/up scheduling where possible;
  • attention window argument;
  • sink-token option;
  • stage timing for norm, projections, attention, and logits;
  • full-context attention as a correctness/control path.

The target is not to prove every marketing claim at once. It is to find the smallest honest K3-native model shape where the sparse-int4 hardware, runtime scheduling, and attention design all point in the same direction.
