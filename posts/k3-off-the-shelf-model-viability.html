K3 model notes

Off-the-shelf model viability on K3

The K3 can run public language-model artifacts, and some of them are genuinely useful development targets. The practical lesson so far is not "download any 4-bit model and expect the accelerator to save it." It is that model architecture, quant layout, backend path, prompt-vs-decode shape, and runtime stability decide whether a public artifact is worth your time.

The short version

If I had to pick from the public artifacts tested so far, I would rank them this way:

Artifact class Current read Why
Gemma 4 26B A4B UD-Q4_K_M Best current development target Fits better than Q8_0, strong SpacemiT prefill, bounded prompt sanity, reusable harness.
Gemma 4 26B A4B Q8_0 Useful reference path Works and validates the supported-format rule, but is memory-tight and slower than UD-Q4_K_M.
Qwen3 0.6B Q4_0 Good small control Easy to run; upstream SpacemiT strongly improves prompt processing, but it is not a serious large-model proof.
LFM2.5 8B A1B Q4_0 Interesting architecture/runtime fit Fast because attention is sparse across layers and the packed Q4_0 MoE path fits the helper.
Qwen3.6 35B A3B UD-Q4_K_M Promising boundary target Loads and shows SpacemiT gains, but benchmark stability and output-quality evidence are not ready.
RedHatAI Sparse Llama W4A16/2:4 Research input, not a chat artifact yet Real sparse weights map to K3 masks and raw tiles, but full inference is not implemented.
Gemma 4 26B A4B UD-IQ4_XS Poor first target Hot expert tensors use IQ formats the current SpacemiT path does not accelerate.

The selection rule

The most reusable rule is simple: inspect expert tensor quantization before downloading or benchmarking huge files.

For the current upstream SpacemiT llama.cpp path, useful public GGUF candidates are the ones whose hot expert tensors land in formats the backend can actually accelerate. The current evidence favors Q4_0, Q4_K, Q5_1, Q5_K, and Q8_0 in the right shapes. It does not favor Q5_0 as a first target, and it does not favor IQ-heavy expert layouts such as the first Gemma UD-IQ4_XS pass.

That is why the best-looking artifact was not the smallest one by name. Gemma UD-Q4_K_M uses Q4_K gate/up expert tensors and Q5_1 down expert tensors. Gemma UD-Q4_K_S uses Q5_0 down tensors, which looked bad in the primitive probe. Gemma UD-IQ4_XS uses IQ expert tensors that did not map to the accelerated path.

Small models are controls, not the answer

The first local inventory only had tiny and small artifacts. That was enough to test whether the board could run llama.cpp paths, but not enough to characterize K3 as an LLM board.

Model Quant Prompt rate Generate rate
SmolLM2 135M Q4_K_M 88.457 tok/s 54.975 tok/s
Qwen3 0.6B Q4_0 22.574 tok/s 15.812 tok/s

Those are useful bounded checks, but they are not evidence for 30B-class viability. The 0.6B Qwen run proved that a public Qwen GGUF could be downloaded, checksummed, loaded, and benchmarked. It did not prove the datasheet's 30B claim, stock IME2 dispatch, FP4 sparse runtime support, or TCM-resident model tensors.

Qwen3 Q4_0 showed the upstream path

A current upstream llama.cpp SpacemiT build made Qwen3 Q4_0 more interesting. Compared against a matched upstream generic RVV build from the same source tree, SpacemiT strongly improved prompt processing but barely moved single-token generation.

Qwen3 0.6B Q4_0 SpacemiT Generic RVV Ratio
pp128 330 tok/s 98 tok/s about 3.35x
pp512 250-253 tok/s 90 tok/s about 2.79x
tg128 33.2 tok/s 32.0 tok/s about 1.04x

That pattern repeats across the investigation: K3 has a much easier time accelerating batched prompt work than batch-one decode. Once Q4_0 projection work moves off the hot path, attention and other residual work become visible.

LFM2.5 was a model-shape lesson

LFM2.5 8B A1B Q4_0 ran locally at 9.637 prompt tok/s and 7.828 generate tok/s in the early installed-backend run. Later runtime inspection showed why it was worth studying: its architecture is friendlier to this board than a plain every-layer-attention transformer.

  • Only 6 of 24 layers are attention layers.
  • Most layers use shortconv/mixer blocks rather than full attention.
  • The public Q4_0 GGUF expert tensors can use the SpacemiT MUL_MAT_ID helper in the patched upstream path.

With the opt-in expert path, LFM prompt processing moved from about 38.9 tok/s to about 82.2 tok/s. Decode barely changed until the Q6_K tied token embedding/output projection was converted to Q4_0, which moved decode from about 15.3 tok/s to about 18.0 tok/s.

The useful conclusion is not "LFM is magically fast." It is that model architecture and packed expert-op layout matter. A synthetic sparse-projection skeleton that treats active experts as separate calls does not match the real llama.cpp MoE dataflow.

Gemma 4 26B A4B became the practical target

Gemma 4 26B A4B was the first model family where the investigation turned into a practical public-artifact selection workflow.

The first target, UD-IQ4_XS, was the wrong fit. It loaded and ran through generic RVV, but its hot expert tensors were mostly IQ4_NL and IQ3_S. The current SpacemiT backend did not accelerate those expert formats, so the SpacemiT path was much slower than generic RVV on small prefill.

Header inspection then pointed to two better choices:

  • Q8_0: all hot expert tensors in a supported format, but a 25 GiB model footprint and high memory pressure.
  • UD-Q4_K_M: Q4_K gate/up experts and Q5_1 down experts, much smaller than Q8_0, and aligned with the primitive evidence.

Gemma Q8_0 worked, but was not the winner

The Q8_0 Gemma artifact confirmed the selection rule. It was faster through SpacemiT than generic RVV in the checked envelope, but it was memory-tight on the 31 GiB board.

Gemma Q8_0 Generic RVV SpacemiT Speedup
pp128 10.40 tok/s 17.67 tok/s 1.70x
tg16 4.10 tok/s 5.52 tok/s 1.35x
chat sanity 9.5 prompt / 3.8 gen 15.3 prompt / 4.5 gen clean short run

Q8_0 remains a useful reference path. It is not the best first development target because it is large, leaves less headroom, and loses to UD-Q4_K_M on the fixed prompt set.

Gemma UD-Q4_K_M is the current best candidate

UD-Q4_K_M is the strongest off-the-shelf model candidate checked so far. It is smaller than Q8_0, uses the right expert quant layout for current SpacemiT support, and has both throughput and bounded prompt-sanity evidence.

Gemma UD-Q4_K_M Generic RVV SpacemiT Speedup
pp128 10.11 tok/s 39.87 tok/s 3.94x
tg16 4.38 tok/s 6.00 tok/s 1.37x
fixed prompt average 9.80 prompt / 3.80 gen 36.78 prompt / 4.56 gen 3.6-4.0x prompt, about 1.2x gen

The fixed prompt harness is important. It does not prove broad model quality, but it is more useful than only reporting llama-bench. Five practical prompts completed through SpacemiT with rc 0 and released TCM afterward. Q8_0 completed the same prompts too, but averaged 17.18 prompt tok/s and 4.12 generation tok/s versus UD-Q4_K_M's 36.78 and 4.56.

A small rubric did not find a bounded quality reason to prefer Q8_0 over UD-Q4_K_M. Both passed the target-selection prompt when given the measured board facts, and both were otherwise limited by answer caps or imprecise wording. That leaves UD-Q4_K_M as the better current development target on performance, memory, and layout grounds.

Qwen3.6 35B A3B is promising but not ready

Qwen3.6 35B A3B UD-Q4_K_M is a viable development target, not a recommendation. It loads in current upstream llama.cpp, fits the board better than Q8_0 would, and shows useful SpacemiT acceleration in one-row benchmark runs.

Qwen3.6 35B A3B Generic RVV SpacemiT Ratio
pp64 6.48 tok/s 22.92 tok/s 3.54x
pp128 9.17 tok/s 28.08 tok/s 3.06x
tg16 3.61 tok/s 5.78 tok/s 1.60x

The boundary matters: combined benchmark processes can abort, simple smoke prompts behaved poorly in the checked build, multimodal behavior was not tested, and low-level Q5_K/Q6_K correctness remains separate backend research. This is a strong lead for future work, not something I would hand to a user as the current practical choice.

Sparse Llama is not off-the-shelf yet

The RedHatAI Sparse Llama 3.1 8B W4A16/2:4 artifact is useful for research because it is shaped toward the K3's sparse-native peak path. Its Marlin metadata decodes cleanly, and a real layer-0 sparse tile can be mapped to SpacemiT recovery masks and matched against the raw vmadot.sp i4 instruction.

That is excellent kernel/projection evidence. It is not full W4A16 sparse Llama inference. There is no token throughput claim, stock placement, TCM residency proof, runtime activation quantization path, attention/KV scheduling, or model-quality proof yet.

What this says about buying or using K3

The board is now past "can it run a model?" The better question is "which public model artifacts align with the runtime that exists today?"

Current answer:

  • Use small Qwen/SmolLM-style artifacts as controls and smoke tests.
  • Use Gemma 4 26B A4B UD-Q4_K_M as the current serious development target.
  • Use Q8_0 as a reference when memory pressure is acceptable.
  • Use Qwen3.6 35B A3B UD-Q4_K_M as a research target, not a user recommendation.
  • Use sparse-native Llama artifacts to build the next runtime, not as current chat artifacts.
  • Avoid IQ-heavy or Q5_0-heavy expert layouts as first targets unless the goal is backend research.

The publication boundary

I would not publish any of these as a default public model artifact yet. The practical recommendation is narrower: UD-Q4_K_M is the best current development target we have tested, and the workflow around it is reusable.

A stronger recommendation needs longer task-specific quality checks, more context-length coverage, clean one-row benchmark discipline, TCM cleanup checks, and preserved generic-RVV versus SpacemiT comparisons. The useful progress is that we now know what to inspect before wasting time: expert quant layout, prompt/decode split, backend attribution, and whether the model's architecture creates enough batched work for the K3 path to matter.

No analytics, comments, or tracking scripts are used here.
