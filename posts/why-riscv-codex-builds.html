Project rationale

Why build Codex for RISC-V?

RISC-V development boards are real computers now, but the software ecosystem still has rough edges. This project is about making one useful tool available where it is most needed: on the machine being debugged.

The problem

Working on RISC-V Linux often means diagnosing missing packages, incomplete release matrices, architecture assumptions, cross-compilation gaps, and runtime dependencies that quietly only exist for x86_64 and arm64. That is exactly the kind of work where a coding agent can help: inspect logs, patch source, rebuild, test, and keep track of the boring details.

The catch is that many coding agents are themselves difficult to run on less common architectures. Gemini CLI is a Node.js and TypeScript project. OpenCode’s current package metadata and development workflow are Bun-centered, and its published optional binaries cover common x64/arm64 targets. Claude Code now has native installers, but its documented npm/platform support is also centered on x64 and arm64. These tools may improve, and some may already be possible with enough effort, but the target moves quickly.

Why Codex

Codex CLI was the most practical target here. The current project is overwhelmingly Rust, and OpenAI publishes native release binaries for mainstream platforms. RISC-V was not just a matter of running cargo build, because code mode depends on V8 through rusty_v8, and the Linux sandbox needed a RISC-V seccomp architecture selector. Still, the dependency shape was understandable and fixable.

That made Codex a good fit for this experiment: build the missing RISC-V pieces, keep the patch small, publish the artifact, and document the process clearly enough that future builds do not start from zero.

What this is not

This is not a claim that Codex is the only agent worth using on RISC-V. Goose deserves an honorable mention: it was one of the easier projects to target in earlier experiments, and its MCP-centered design is a reasonable bet for interoperability. For my specific goal, though, the extra extension layer made the direct build-fix-test loop feel less straightforward, and getting state-of-the-art hosted models into that workflow was less direct. Local models were also not quite strong enough at the time for the self-diagnosis loop RISC-V development tends to need. Your mileage may vary.

Goal

The goal is simple: make it easier to run a capable coding agent directly on RISC-V Linux boards, starting with the Orange Pi RV2 / SpacemiT K1 class of hardware.
