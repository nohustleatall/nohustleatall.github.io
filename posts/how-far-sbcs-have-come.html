Board comparison

How far have small boards come?

A cross-board bring-up pass is a good reminder that small boards have improved a lot, but not in one simple direction. The Orange Pi RV2 streams well and has real RVV capability. The Pi 400 is still a stronger general CPU. Older boards show how much the floor has moved.

The boards

The comparable run used the same bring-up pack across current boards and a workstation reference: Orange Pi RV2 / Ky X1, Raspberry Pi 400, Raspberry Pi 3 class hardware, Raspberry Pi Zero W class hardware, and an x86_64 reference machine. Intel Edison was used only as historical color because the current Python runner could not run unchanged on its old software stack.

One table, many stories

A flat benchmark table does not produce one winner. It shows workload shape. The Pi 400 wins most general CPU work. RV2 is competitive in streaming memory and fixed binary gateway work. Pi 3 remains a useful middle baseline. Pi Zero W is the historical low-end floor.

Compiler maturity matters

The compiler flag sweep explains one of the biggest traps. On RV2, ordinary -O3 C stayed around 0.19-0.20 GMAC/s for the int8 dot loop. The explicit RVV version reached about 2.3 GMAC/s. That is a real vector win, but only when the code actually uses RVV-shaped kernels.

The ARM boards got more help from the compiler. Pi 3 moved from about 0.23 GMAC/s at -O2 to about 1.00 GMAC/s at -O3, close to its explicit NEON result. Pi 400 moved from about 0.85 GMAC/s at -O2 to about 2.24 GMAC/s at -O3, while explicit NEON reached about 5.0 GMAC/s.

That does not mean NEON is always better than RVV. It means the mature ARM compiler path is doing useful work automatically on this pattern, while the RV2 path still needs explicit kernels.

Memory explains the split

The memory working-set sweep is the best explanation for why the RV2 can look good on streaming work and weak on pointer-heavy code. At large working sets, RV2 sequential reads are close to Pi 400, and RV2 sequential writes are strong. Pointer chasing is different.

64 MiB working set RV2 Pi 400 Pi 3 Pi Zero W
Sequential read 4.20 GB/s 4.29 GB/s 2.21 GB/s 0.49 GB/s
Sequential write 5.64 GB/s 3.19 GB/s 1.74 GB/s 0.41 GB/s
Pointer chase 2.66 M steps/s 6.57 M steps/s 5.61 M steps/s 3.28 M steps/s

That table captures the RV2 story well. Streaming bandwidth is fine. Dependent pointer latency is not. So contiguous buffers, fixed frames, and regular kernels are good candidates. Hash tables, object graphs, branchy parsers, and allocator-heavy runtimes are worse fits.

The appliance result

The framed binary gateway result is where the RV2 looks most practical outside ML. It tied the Pi 400 and stayed close to the x86 reference because the workload is linear: validate framed records, compute CRC, filter, and copy payloads. When the pipeline became text redaction, scalar audio, or JSON-ish formatting, Pi 400 pulled away.

That is a useful niche, not a universal claim. RV2 looks credible for fixed binary stream gateways, simple validators, binary log forwarding, and checksum/filter/copy appliances. It is less compelling for text-heavy transformation.

The historical floor

The Pi Zero W result is useful because it shows how far the low end has moved. It is far behind newer boards on memory bandwidth, scalar throughput, vector-style arithmetic, and appliance pipelines. But it is also a warning: in the 64 MiB pointer-chase result, the Pi Zero W was close to or above the RV2. Old does not always mean slower on every latency-shaped primitive.

The Intel Edison is even more historical. Its mature x86 lineage did not make an old embedded Atom-class board competitive with newer ARM and RISC-V boards. The software stack itself became part of the benchmark problem: old kernel, old userland, no Python 3 in the tested install, and toolchain caveats. Generation and software age matter as much as ISA branding.

What changed over time

Compared with older boards, current small boards are no longer just GPIO toys. They can run meaningful local services, stream tens of megabytes per second through application code, run vector kernels, and host real development loops. The RV2 can run Codex, package opencode, do local STT experiments, and run MiniLM-style embedding workloads.

But the comparison also makes the limit clear. Modern SBC progress is not a single ladder. Scalar cores, compiler maturity, memory latency, vector intrinsics, ONNX operator coverage, and workload shape all move at different speeds.

Practical takeaway

If you want a generally faster small Linux computer, the Pi 400 remains a strong baseline. If you want to learn where RISC-V Vector work pays, build portable RVV experiments, or run fixed-format stream appliances, the RV2 is genuinely interesting.

The fun part is not that one board wins everything. The fun part is that small boards have become capable enough that the differences are now specific: branch latency, pointer chasing, auto-vectorization, quantized kernels, streaming buffers, and real application shape. That is progress.
