Performance notes

How many threads should you use on Orange Pi RV2?

The current answer is: do not default to all cores. On Orange Pi RV2 / Ky X1-class hardware, thread count is a real benchmark variable, and the best setting depends on the graph and the complete application pipeline.

Short version

Start full single-stream workloads at 2 threads, try 4 threads when the hot path is clearly a larger ONNX or GEMM-heavy section, and prove any all-core setting before using it. In the current tests, 8 threads often hurt latency once preprocessing, decoding, or decoder-side system time entered the workload.

Why this matters

Small-board inference is not just one operator running in isolation. A real speech-to-text path may include audio loading or capture, feature extraction, ONNX inference, greedy or beam decoding, a language model, endpointing, UI work, shell interaction, and logging. If the model uses every core, the rest of the application still has to run somewhere.

That is why isolated operator timing and full-pipeline timing can disagree. The right thread count is the one that wins the workload you are actually shipping or using, not the one that wins a single kernel benchmark.

ONNX synthetic matrix

The first sweep used default ONNX Runtime 1.18.0 with the CPU provider. Each setting ran prebuilt synthetic ONNX probes with nine measured runs after warmup. Times are median milliseconds.

Probe 1 thread 2 threads 4 threads 8 threads Best
gemm_fp32_64 0.440 0.489 0.205 0.186 8
gemm_fp32_128 0.925 0.752 0.497 0.465 8
gemm_fp32_medium 2.389 1.926 1.633 5.598 4
matmul_integer_u8s8_medium 1.349 0.741 0.763 0.943 2
conv_fp32_medium 2.694 1.742 5.119 6.192 2
qlinear_conv_u8s8_medium 1.702 1.257 1.538 2.127 2

The synthetic result is already enough to reject one global answer. Small fp32 GEMM benefited from 8 threads. Medium fp32 GEMM peaked at 4 and regressed badly at 8. Medium int8 MatMul and conv-like probes favored 2 threads.

Scratch CTC full pipeline

The more important test used the scratch 384x10 SentencePiece CTC speech model, exported as MatMul/Gemm dynamic int8 ONNX. This benchmark included feature extraction, fixed-window ONNX chunks, greedy decode, and beam4/top16 plus token-bigram-LM decode across three audio samples.

Threads Median greedy RTF Median beam+LM RTF Median ONNX s/sample Max beam+LM RTF
1 0.243 0.318 2.682 0.337
2 0.160 0.232 1.814 0.253
4 0.172 0.241 1.653 0.269
8 0.249 0.320 2.556 0.325

Here the full pipeline favored 2 threads. Four threads slightly reduced median ONNX time, but not enough to beat 2 threads after decode and the rest of the pipeline were included. Eight threads regressed close to the 1-thread result.

Vosk recap

The older Vosk runs show the same practical pattern. Both open-vocabulary mode and constrained grammar mode favored 2 threads. Eight threads was worst in both modes.

Vosk mode 1 thread RTF 2 threads RTF 4 threads RTF 8 threads RTF
Open vocabulary 1.595 1.465 1.689 2.210
Grammar mode 0.452 0.274 0.303 0.507

Grammar mode is much faster because it solves a constrained problem. It is useful for command interfaces, not a replacement for open-vocabulary dictation. The thread lesson is separate: even when the task is faster, all cores were still the wrong default.

Practical guidance

  1. Run the isolated model or operator at 1, 2, 4, and 8 threads.
  2. Run the complete pipeline at the same thread counts.
  3. Report median and max or p95-like tail behavior, not only the best run.
  4. Prefer the setting that wins the full workload, not the isolated kernel.
  5. Leave at least one or two cores worth of scheduling headroom for interactive or streaming applications.

For current RV2 single-stream inference work, 2 threads is the best first guess. Four threads is worth trying for larger ONNX-heavy sections. Eight threads should be treated as something to justify with measurements, not as the natural setting for an 8-core board.

No analytics, comments, or tracking scripts are used here.
