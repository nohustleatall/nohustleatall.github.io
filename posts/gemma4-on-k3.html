K3 model notes

Gemma 4 on K3: dense, MoE, and MTP are three different stories

Gemma 4 has been useful in the K3 investigation because it forces several separate questions apart. A dense 12B model tests the plain modern-transformer speed tax. The 26B A4B MoE tests whether expert tensor layout matches the SpacemiT path. The E2B/E4B QAT GGUFs test Google's smaller edge shapes. MTP tests whether speculative decoding can improve decode once the assistant is cheap enough and the runtime avoids unnecessary logits.

The short version

The best Gemma result so far is not the dense 12B model and not the largest Q8_0 artifact. It is Gemma 4 26B A4B UD-Q4_K_M, because its hot expert tensors line up with formats the current SpacemiT llama.cpp path can accelerate.

Gemma path Current K3 read Why
Gemma 4 12B IT Q4_0 Useful dense baseline, poor speed fit About 24.70 tok/s prompt and 3.38 tok/s decode in the checked bench path.
Gemma 4 26B A4B UD-IQ4_XS Wrong first MoE target Hot experts are mostly IQ formats that do not map to the current SpacemiT expert path.
Gemma 4 26B A4B Q8_0 Good reference, memory-tight Real SpacemiT win, but close to the 31 GiB board's practical memory edge.
Gemma 4 26B A4B UD-Q4_K_M Best current Gemma development target Q4_K gate/up plus Q5_1 down experts give strong prefill acceleration and manageable size.
Gemma 4 MTP assistant Prototype speed win on E2B/E4B, not production-ready yet KV mapping, Q4 assistant bodies, ordered tensors, and process-only updates now work in a short direct harness.

Dense 12B is the reality check

Gemma 4 12B IT Q4_0 is valuable because it answers a boring but important question: what happens when a current dense Gemma model is run through the checked K3 SpacemiT llama.cpp path?

Model Size Params pp128 tg32 tg128
Gemma 4 12B IT Q4_0 6.26 GiB 11.91B 24.70 tok/s 3.38 tok/s 3.38 tok/s
LFM2 24B A2B Q4_0 12.54 GiB 23.84B 71.37 tok/s 16.70 tok/s 16.83 tok/s

That comparison is the point. LFM2 has roughly twice the total parameters of the dense Gemma 12B artifact, but it decodes much faster on the checked K3 path. Parameter count is not the speed predictor here. Active compute, attention frequency, tensor layout, and the packed expert runtime path matter more.

Gemma 4 12B may still be useful if the model quality is what you need and the speed is acceptable. It is not the shape that teaches the board's best-case acceleration story.

The smaller QAT targets are better speed shapes

The later E2B and E4B QAT GGUF tests changed the Gemma story. These are text-only runnable GGUF targets, not the separate mobile compressed-tensors releases. They use smaller hidden sizes, low KV-head counts, sliding-window attention, and model shapes that look much more appropriate for K3 decode than the dense 12B baseline.

QAT target Params Size Shape Target-only decode
Gemma 4 E2B QAT Q4 4.63B 3.10 GiB 35 layers, hidden 1536, KV heads 1, window 512 about 10.9-12.0 tok/s
Gemma 4 E4B QAT Q4 7.46B 4.79 GiB 42 layers, hidden 2560, KV heads 2, window 512 about 6.6-7.2 tok/s
Gemma 4 12B Q4_0 11.91B 6.26 GiB dense 12B-class text model about 3.38 tok/s

That is an important correction to the simple dense-vs-MoE framing. Gemma 4's small QAT text models are not MoE like the 26B A4B target, but they still fit the K3 better than the 12B dense artifact because their attention and memory footprint are more edge-shaped.

The first 26B MoE target was the wrong quant

Gemma 4 26B A4B is more interesting architecturally, but MoE alone is not enough. The first small-ish public target, UD-IQ4_XS, loaded in current upstream llama.cpp and ran through generic RVV, but its hot expert tensors were a poor match for the SpacemiT backend.

The checked layout had 30 layers, hidden size 2816, 128 experts, and 8 active experts. The important part was the tensor format:

  • layers 0 through 28 expert down tensors: IQ4_NL;
  • layers 0 through 28 expert gate/up tensors: IQ3_S;
  • the final layer differed, but most hot expert work was still IQ-heavy.

Generic RVV was the usable baseline for that artifact: about 9.31 tok/s at pp128 and 4.37 tok/s at tg16. The SpacemiT path could complete small prefill rows after clearing stale TCM blocks, but it was around 1.1 tok/s for pp8 through pp32. That is a useful negative result: placing the wrong quant layout on the vendor path can make things worse.

The primitive probe gave a selection rule

The next step was not "download every Gemma quant." It was to test which expert-like tensor formats actually get useful acceleration.

Format Gate/up-like result Decode-like result Practical read
Q4_0 large speedup large speedup Best clean path, but not always the available useful artifact.
Q4_K 51.43x 17.73x Good gate/up candidate.
Q5_1 48.79x 16.23x Good down-projection candidate.
Q5_K 76.41x 31.05x Promising where present.
Q5_0 about 1.00x about 1.00x Avoid as a first target despite nominal support.
IQ4_NL / IQ3_S no useful SpacemiT support in the checked path no useful SpacemiT support in the checked path Good RVV baseline or backend research target, not the first accelerated Gemma target.

That produced the durable triage rule: inspect expert tensor layout before investing board time. For Gemma 4 26B A4B, UD-Q4_K_M became more attractive than UD-Q4_K_S because the former uses Q5_1 expert down tensors while the latter uses Q5_0 down tensors.

Q8_0 proved the MoE path can work

The Q8_0 artifact was a useful reference because all hot expert tensors were in a supported format. It was also large: llama.cpp reported about 25.00 GiB, and the first RVV load probe reached roughly 31.5 GB RSS on the checked 31 GiB board.

Gemma 26B A4B Q8_0 Generic RVV SpacemiT Speedup
pp128 10.40 tok/s 17.67 tok/s 1.70x
tg16 4.10 tok/s 5.52 tok/s 1.35x
chat sanity 9.5 prompt / 3.8 gen 15.3 prompt / 4.5 gen clean short run

The Q8_0 result matters because it confirms the selection rule at model level. When Gemma's hot expert tensors use a format the backend accelerates, SpacemiT can win. The downside is memory pressure, not lack of acceleration.

UD-Q4_K_M is the practical Gemma target

UD-Q4_K_M was the better tradeoff. It is much smaller than Q8_0, keeps the hot expert tensors in useful formats, and produced the strongest Gemma result so far.

Gemma 26B A4B UD-Q4_K_M Generic RVV SpacemiT Speedup
pp64 10.14 tok/s 39.27 tok/s 3.87x
pp128 10.11 tok/s 39.87 tok/s 3.94x
tg16 4.38 tok/s 6.00 tok/s 1.37x
fixed prompt average 9.80 prompt / 3.80 gen 36.78 prompt / 4.56 gen about 3.75x prompt, 1.20x gen

The perf profile matched the tensor-layout story. The SpacemiT prefill run sampled MoE kernels for i8/i4 and i8/i5 work, consistent with Q4_K gate/up and Q5_1 down experts. Repack still appears in process-level profiles because tensors are converted when loaded into the SpacemiT buffer, but the repeated prefill profile showed steady-state IME work rather than a pure repack artifact.

The operating boundary is also clearer now. Eight threads are the current safe setting for this path. Sixteen threads aborted because thread IDs exceeded the backend's preferred-core list. Combined multi-row llama-bench sweeps hit allocator failures, so the reliable harness runs one row per process and checks TCM release afterward.

Quality evidence is bounded

The fixed-prompt harness is not a broad model-quality benchmark. It is still better than a single "hello world" generation. It ran target selection, quant-layout explanation, parser, risk-register, and runbook prompts, saving timing and completion metadata for each row.

On the identical five-prompt set, UD-Q4_K_M averaged 36.78 prompt tok/s and 4.56 generation tok/s through SpacemiT. Q8_0 averaged 17.18 prompt tok/s and 4.12 generation tok/s. The small rubric did not identify a quality reason to prefer Q8_0 over UD-Q4_K_M; both passed target selection when given measured board facts and were otherwise mostly partial because of answer caps or imprecise wording.

That is enough to use UD-Q4_K_M as a development target. It is not enough to publish a general artifact recommendation or claim broad Gemma quality on K3.

MTP is not just another GGUF

Gemma 4's MTP assistant is interesting because it could attack the decode bottleneck directly. But it is not a simple matter of loading a second model file into the current runtime.

The official assistant path works in development Transformers with a matching Gemma 4 target. A tiny CPU assisted-generation run completed and produced the same first four generated tokens as the baseline, with the MTP path taking 29.519 seconds versus 36.158 seconds in that particular reference run. That proves the mechanism exists in the official software stack.

The assistant is a real drafter, not a normal standalone language model. For the 12B QAT assistant, the checked metadata showed:

  • about 422.9M parameters in BF16;
  • 4 assistant layers;
  • assistant hidden size 1024;
  • backbone hidden size 3840;
  • sliding/sliding/sliding/full attention layer pattern;
  • input projection over token embedding plus target hidden state.

Direct token-ID inference fails by design: the assistant needs inputs_embeds and shared_kv_states. In Google's path, the target model returns hidden states and shared KV state, then the assistant proposes draft tokens, and the target verifies them in parallel.

Why native llama.cpp MTP is a real project

Current upstream llama.cpp has speculative/MTP machinery, but it is shaped around NextN-style draft heads. Gemma 4 is different. The assistant needs target token embedding, target last hidden state, and target full/SWA KV state. Existing hidden-row plumbing is not enough.

A local prototype made real progress:

  • the upstream converter originally rejected Gemma4UnifiedAssistantForCausalLM;
  • a prototype converter emitted a gemma4-assistant GGUF;
  • the assistant GGUF had 49 tensors and was about 822 MiB in BF16;
  • target hidden-state extraction was prototyped;
  • target input-embedding extraction was prototyped;
  • assistant attention over target full/SWA KV was prototyped in local CPU smoke tests.

The first 12B QAT prototype moved the blocker from "unsupported architecture" to "match the recurrence exactly." One-token draft mode accepted 5 out of 5 draft tokens in the local constrained test. Two-token draft mode was stable but accepted 5 out of 10, exposing that second and later draft steps need exact recurrence behavior.

E2B and E4B make MTP concrete

The newer E2B and E4B QAT MTP probes made the speculative-decoding story much sharper. The key runtime fix was durable target-KV mapping: assistant sliding-attention layers map to the target's last real sliding KV layer, and assistant full-attention layers map to the target's last real full KV layer. For the checked targets that means E2B maps to layers 13 and 14, E4B maps to 22 and 23, and 12B maps to 46 and 47.

After that mapping, E2B and E4B MTP run without the previous target-KV crash. The BF16 full-vocabulary assistant fallback is still too slow, but Q4 assistants start to help.

Target Target-only Dense Q4 assistant MTP Best prototype MTP
E2B QAT Q4 10.88 tok/s 10.61 tok/s, depth 2 13.54 tok/s with mixed sparse assistant
E4B QAT Q4 6.59 tok/s 8.04 tok/s, depth 2 9.32 tok/s with mixed sparse assistant

Those are short direct-harness results, not a polished llama.cpp feature and not a production claim. They matter because they show MTP can produce an end-to-end speedup on K3-class Gemma QAT targets when the assistant body is cheap and the runtime avoids wasted vocabulary projections.

A later real E4B sweep made the boundary sharper. The ordered sparse rowstore scorer works and is cheap, but the full speculative callback still did not beat the target verification baseline. The best easy-prompt callback row reached 0.480 tok/s with perfect acceptance, while width-4 target verification reached about 0.546 token-equivalent/s. Removing full-vocabulary logits from target verification did not fix it: width-8 no-logits verification was still only about 0.553 token-equivalent/s. That means the current MTP blocker is target verification, not candidate scoring or logits materialization.

The ordered sparse path is the real MTP trick

The official E2B/E4B assistants are not meant to score the full 262144-token vocabulary every draft step. They use ordered embeddings: centroid vectors plus token ordering metadata. The runtime scores centroid buckets, expands the selected buckets into candidate token ids, and scores only that candidate subset.

The first converter dropped the ordered tensors, which forced a full-vocabulary fallback. A follow-up converter now preserves and loads them:

  • assist.centroids.weight, BF16, shape [256, 2048];
  • assist.token_ordering.weight, I32, shape [262144].

The sparse candidate correctness probe is encouraging. For E2B, the top 32 centroid buckets included the dense greedy token 29 out of 32 times, and top 64 included it 31 out of 32 times. For E4B, top 32 also hit 29 out of 32, top 64 hit 30 out of 32, and top 128 hit 32 out of 32. That confirms the bucket-contiguous token ordering interpretation and gives a concrete candidate-count tradeoff.

The current sparse implementation is still not the final fast path. CPU sparse scoring over 4096 candidates costs about 4.6 to 4.7 ms per assistant step, and larger candidate sets are correspondingly more expensive. Fully quantized sparse scoring also exposed a backend placement problem: dynamic GET_ROWS over a Q6_K output table in the SpacemiT buffer does not schedule cleanly, and forcing it to CPU produced unusable drafts. The useful workaround so far is a mixed assistant: Q4_0 transformer/projection body, BF16 output table, sparse scoring, and a process-only path that skips logits when the assistant only needs to update hidden state.

The mobile QAT releases are design evidence

The mobile QAT releases are not immediate drop-in models for the current K3 llama.cpp path. The mobile compressed-tensors format uses packed safetensors with explicit scales, not GGUF Q4_0. But the schema is still highly relevant because it shows what Google's edge recipe is optimizing:

  • small KV footprint and short sliding windows;
  • static activation scales;
  • channel-wise low-bit weights;
  • selective 2-bit compression for memory-heavy embeddings and output heads;
  • 8-bit treatment for sensitive gate/projection paths.

That lines up with the K3 results. Normal GGUF Q4 is runnable today, but the mobile schema suggests the next useful experiment is not "download the mobile model and hope." It is a CT-style packed int4/int8 projection microbenchmark with static activation scales, compared against the current GGUF Q4_0 SpacemiT path.

What to believe today

The Gemma result is conditional but useful:

  • Dense Gemma 4 12B is a good control, not a good K3 speed shape.
  • Gemma 4 E2B/E4B QAT Q4 are much better small Gemma speed shapes for K3.
  • Gemma 4 26B A4B can be a good K3 development target, but only with the right expert tensor layout.
  • UD-IQ4_XS is a negative boundary for current SpacemiT acceleration.
  • Q8_0 proves the accelerated path can work, but it is memory-tight.
  • UD-Q4_K_M is the current practical Gemma target because Q4_K plus Q5_1 aligns with the measured backend behavior.
  • MTP has short-harness prototype wins on E2B and E4B, but the later real E4B callback sweep shows target verification is the blocker before it becomes a practical speed path.

The next useful Gemma work is not another headline benchmark. It is a longer quality harness for UD-Q4_K_M, a clean low-level correctness pass for Q5-family expert paths, and a target-verification shape audit for MTP. Candidate scoring is already cheap enough to stop being the first suspect.
