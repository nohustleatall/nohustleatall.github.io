RISC-V coding agents and the runtime wall

If you are trying to run modern coding agents directly on a RISC-V Linux board, the hard part may not be the agent. It may be the JavaScript runtime and native packaging stack underneath it.

Short version

Codex was practical to publish for the Orange Pi RV2 because its Rust codebase had a small number of concrete RISC-V blockers: a target V8 archive, a cross-compiled OpenSSL path, and a Linux seccomp architecture fix. Bun-based agents are in a different place. The first wall is not application logic. It is that the current standalone executable and JavaScriptCore/WebKit stack do not provide a clean RISC-V Linux path.

Target Observed status Practical meaning

opencode: first blocker

The obvious path for a Bun-centered CLI is to ask Bun to compile a standalone Linux RISC-V executable. That failed immediately: the standalone target matrix did not accept a RISC-V Linux target. The upstream build script also enumerated common Linux, macOS, and Windows targets for x64 and arm64, but not RISC-V.

That is an important negative result. If your plan is simply "add riscv64 to the release target list," expect to hit Bun itself before you get to any meaningful application test.

opencode: second blocker

The next experiment was a more ordinary JavaScript bundle that could potentially run under a different runtime. That also did not become a clean path. Parts of the CLI and TUI graph still imported Bun builtins, and dependency resolution exposed native/package assumptions that would need real porting work rather than a packaging tweak.

This does not mean opencode can never work on RISC-V. It means the useful work is probably not at the final packaging layer. A better starting point is to split the runtime-dependent parts from the agent logic and make an explicit supported mode for a non-standalone, non-Bun executable environment.

Bun: what got farther

Bun itself was more interesting. A RISC-V Linux configuration could get through parts of the build system, and a direct Zig object build from the source root succeeded for the tested revision. That is progress, but it is easy to overread: a cached RISC-V object is not the same thing as a usable Bun runtime.

The larger C++ build reached hundreds of build steps, including RISC-V-specific TinyCC objects and networking/runtime pieces, before failing in the JavaScriptCore/WebKit area. The failures were not a single missing flag. They involved platform allocator selection, ICU/header plumbing, and JavaScriptCore source/header assumptions that showed up during a cross build.

WebKit/JSC is the real boundary

The most useful lesson is that Bun-on-RISC-V is largely a JavaScriptCore/WebKit porting problem. The experiments reached generated JavaScriptCore sources and JSC shell-related build steps before failing on missing or mismatched internal types and platform configuration details. That is the kind of problem where a small downstream patch may get one build farther, but a durable solution needs upstream-quality architecture support.

If you are trying this yourself, do not start by debugging the coding agent. First prove that your Bun or JSC runtime path is real on RISC-V Linux. Until that is true, every agent-level failure may just be a runtime-port failure wearing a different name.

What I would try next

  • Track Bun's supported executable target matrix before spending time on downstream CLI packaging.
  • Build and test the smallest possible JavaScriptCore shell for RISC-V Linux before attempting the full Bun binary.
  • Keep sysroot, ICU, allocator, and libc choices explicit; hidden host includes can waste days.
  • For Bun-centered agents, separate "can the app bundle" from "can Bun run on the target." They are different milestones.
  • For immediate use on RV2-class boards, start with agent stacks whose native runtime already has a plausible RISC-V story.

Why this matters

RISC-V boards are good places to use coding agents because the work is full of logs, patches, build flags, and repeated smoke tests. But an agent is only as portable as its runtime stack. For now, the Rust/V8 path used by Codex has been easier to turn into a published RV2 artifact than the Bun/JSC path used by some other tools.

That may change. When it does, the fastest route will probably come from treating runtime support as the main project, not as a final packaging chore.
