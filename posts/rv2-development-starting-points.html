Board guide

Orange Pi RV2 development starting points

The Orange Pi RV2 is a useful RISC-V Linux development board, but it rewards a different workflow than x86_64 or arm64. Start by proving the board, runtime, graph form, and package stack before judging the application.

The baseline assumption

Treat the CPU and RVV path as the main programmable target. The board is good for RISC-V Linux work, CPU inference experiments, packaging tests, and RVV kernel exploration. Do not assume that AI marketing numbers imply a generally programmable NPU path until you can point at a device, driver, runtime, and tiny smoke test that proves it.

The practical starting model is simple: Linux userspace works well enough for normal C/C++ and Python experiments, RVV is the interesting acceleration surface, and vendor ONNX Runtime packages are relevant mostly because they can change operator coverage and CPU/RVV behavior.

Record the board first

Before debugging an app or model, capture the basic environment. On small boards, the OS image, kernel, Python version, compiler, and storage situation can change the result.

uname -a
lscpu
python3 --version
gcc --version
clang --version || true
free -h
df -h
python3 - <<'PY'
import platform
print(platform.machine())
print(platform.platform())
PY

Keep this information beside benchmark notes. It is much easier to compare results later if the runtime and board state are recorded before the first failure.

Python package reality

Expect fewer prebuilt wheels than on x86_64 or aarch64. Some packages install cleanly, some need long source builds, and some do not have a practical riscv64 path yet. A failed pip install is not the same thing as a failed model.

  • Use a separate virtual environment per experiment.
  • Record Python version, package versions, and install source.
  • Prefer pure-Python packages or packages with known riscv64 wheels.
  • Budget time for C++, Rust, Fortran, protobuf, and numerical package builds.
  • For ONNX work, keep model generation separate from runtime-only environments.

RVV: prove the shape

For performance work, verify that the compiler can build the RVV code shape you intend to use. Do not assume every intrinsic or tuple type is available in the board image's compiler.

The low-level pattern that has worked best is conservative: write a scalar checksum first, add the RVV implementation second, compare checksums for every shape, and only then time it. Record the exact -march and -mabi flags. A binary that runs is not enough if it silently misses the intended vector path.

The most useful compute lesson so far is that signed int8 is the first serious optimization target. Packed int4 is not automatically faster. If the hot loop has to split nibbles, sign-adjust them, create separate activation streams, and reduce multiple products, unpack overhead can dominate the smaller representation.

ONNX Runtime is part of the target

This is the biggest trap. ONNX export does not make a model portable by itself. The ONNX graph form, runtime version, operator coverage, execution provider, linked shared library, graph optimization setting, and thread count are all part of the deployment target.

The first general ONNX Runtime matrix on RV2 used synthetic probes before judging real models. Two runtime lanes were compared: a default ONNX Runtime 1.18.0 lane and a SpacemiT ONNX Runtime 1.24.0+spacemit.a3 lane. The important finding was operator coverage, not a universal speed win.

Probe Default ORT 1.18.0 SpacemiT ORT 1.24 Practical read
MatMulInteger valid valid Dense dynamic-int8 paths are plausible in both lanes.
QLinearMatMul valid valid Static quantized dense paths are also plausible.
ConvInteger unsigned valid valid Plain support claims are not detailed enough.
ConvInteger u8 x s8 load fail valid Common int8 conv exports may need the newer runtime.
ConvInteger s8 x s8 load fail valid Signed quantized conv is a real compatibility boundary.
QLinearConv u8 x s8 valid valid QLinear/QDQ graph forms may be more portable than some ConvInteger exports.

The same split showed up on both tiny and medium convolution probes. That matched earlier real-model failures where some int8 exports were blocked by ConvInteger under the default runtime but runnable under the newer SpacemiT runtime.

Speed was not one-sided

The vendor runtime was not simply faster at everything. In the matrix pass, default ORT was faster on one medium fp32 convolution probe, while SpacemiT ORT was faster on medium int8 MatMul and QLinearConv. The reason to keep the SpacemiT lane is not "always faster"; it is that it closes a real operator-coverage gap.

Medium probe Default ORT 1.18.0 SpacemiT ORT 1.24
fp32 GEMM 3.835 ms 3.487 ms
u8 x s8 MatMulInteger 0.566 ms 0.331 ms
fp32 Conv 1.567 ms 2.443 ms
u8 x s8 ConvInteger load fail 1.981 ms
u8 x s8 QLinearConv 1.740 ms 1.322 ms

Use synthetic probes first

A useful board sanity suite should include small ONNX graphs from several families before a large real model enters the picture: tensor plumbing, fp32 GEMM, fp32 Conv, MatMulInteger, QLinearMatMul, QLinearConv, signed and mixed signed ConvInteger, QDQ, and one small real model from the target domain.

For each probe, record load success, run success, output checksum, warm median runtime, thread count, runtime version, providers, and graph optimization level. That separates three questions that are otherwise easy to mix together: whether the board has enough compute, whether the runtime can execute the graph, and whether the application code handles preprocessing and postprocessing correctly.

Runtime linkage matters

The runtime imported by Python in your shell may not be the runtime loaded by an application binary. If a binary ships or links its own ONNX Runtime, test that lane separately.

ldd ./your_binary | grep -i onnx || true
python3 - <<'PY'
import onnxruntime as ort
print(ort.__version__)
print(ort.get_available_providers())
PY

If the binary requests a specific libonnxruntime.so soname, changing LD_LIBRARY_PATH may not redirect it to the library you expected. Verify with ldd or a controlled smoke test.

Threading is a benchmark variable

More threads are not automatically better. On small CPU boards, extra threads can hurt latency, increase contention, and leave less headroom for the rest of the application. Test at least 1, 2, 4, and all-core settings. Record median time and outliers, not only the best run.

Good first experiments

  1. Build and run a scalar C hello-world plus one simple numerical loop.
  2. Compile a tiny RVV checksum test and compare it against scalar output.
  3. Run a Python package smoke test in a clean virtual environment.
  4. Run a synthetic ONNX operator matrix for the runtime environment you plan to use.
  5. Run one small real model and verify output against a known checksum, text, or classification.
  6. Run the same model with 1, 2, 4, and all-core thread settings.
  7. Only then move to larger models, custom kernels, or application-level debugging.

Common pitfalls

  • Assuming vendor AI claims imply an accessible general-purpose NPU.
  • Treating riscv64 like x86_64 for Python wheels.
  • Judging a model before checking ONNX Runtime operator support.
  • Saying ConvInteger is supported or unsupported without checking dtype form.
  • Assuming int4 is faster than int8 because it is smaller.
  • Ignoring app-linked runtime libraries.
  • Publishing benchmark notes with local paths, hostnames, or raw device logs.

The useful habit is to make failures precise. Prefer load_fail: ConvInteger u8 x s8, wrong_output: tensor checksum mismatch, or valid_slow over "the model is broken." On this board, that discipline saves a lot of time.
