Audio lab notes

Can a cheap RISC-V board do useful local STT?

The short answer is yes, but not with every model and not for every product shape. On Orange Pi RV2-class hardware, the useful local speech-to-text path is less about chasing the newest model and more about matching runtime, decoder cost, and accuracy to the job.

What was tested

The lab pass compared several local speech-to-text paths on the RV2 board class: Vosk, sherpa models, Moonshine tiny, NVIDIA Parakeet variants, Qwen3-ASR, IBM Granite Speech, and a deliberately small CTC model trained from scratch. The goal was not to declare one universal winner. The goal was to find which approaches are actually plausible on a small RISC-V CPU.

Two measurements matter throughout this post. Real-time factor, or RTF, is runtime divided by audio duration. An RTF below 1.0 is faster than realtime. Loaded-model RTF is the more useful number for an appliance or service, because model load time can be amortized after startup.

Baseline: Vosk still matters

Vosk small remains a useful control because it is traditional, local, and dependency-light. Open-vocabulary decoding on the RV2 was correct enough to keep around, but it was not fast enough for live captioning: the best safe decoder setting landed around 1.37 RTF.

Vosk changed character in grammar mode. With a constrained phrase list, the same board reached about 0.27 RTF at the best thread setting. That does not solve open-vocabulary dictation, but it is immediately relevant for a wake phrase, a voice command appliance, or a small domain-specific listener.

Mode Best observed RTF Practical read
Vosk open vocabulary ~1.37 Useful baseline, not live enough for open captioning.
Vosk constrained grammar ~0.27 Good fit for command mode and appliance-style controls.

The shortlist

The most useful near-term open-vocabulary candidates were the ones that combined a clean runtime path with acceptable loaded-model speed. Moonshine tiny became the best speed/accuracy reference once a bad ONNX Runtime mismatch was fixed. Parakeet realtime EOU and Parakeet TDT also became valid candidates after runner and artifact issues were separated from model quality.

Runtime bugs can look like bad models

One of the more important lessons was not about architecture at all. Moonshine tiny initially looked broken: repeated or gibberish tokens and no clean end-of-sequence behavior. The cause was not the model. The native runner was linked against a source-built ONNX Runtime that produced different encoder tensors from the Python wheel runtime. Relinking to the known-good runtime made checksums match and produced correct text.

The same pattern showed up elsewhere. A Parakeet realtime runner bug took the wrong logits slice from a 4D decoder output. Some sherpa paths needed a soname-compatible ONNX Runtime shim. A mixed int4/int8 Parakeet TDT package blanked out while the official int8 artifacts decoded correctly. The practical rule is simple: bucket failures before discarding a model. Tensor mismatch, tokenizer mismatch, frontend mismatch, decoder-loop bugs, and bad quantization artifacts are different problems.

Int8 beat int4 here

It is tempting to assume packed int4 will make everything faster on a small board. The RV2 low-bit GEMM probes did not support that. The useful hot path was RVV int8 dot product. Int4 values expanded to int8 computed at roughly int8 speed, but keeping weights packed and unpacking them inside the hot loop lost time.

Kernel path Observed result Meaning
RVV int8 dot ~4.2-4.9 GOPS across tested ASR-like shapes Best practical target for hot compute.
Int4 pre-expanded to int8 About int8 speed Useful as a storage/load-time idea, not a compute win.
Packed int4 direct RVV dot ~1.7-1.9 GOPS Correct, but much slower than int8.
Packed int4 unpack-row RVV Often slower, especially for joiner-like shapes Unpack overhead dominates too easily.

For this board, int8 is the thing to design around. Packed int4 may still matter for storage, load time, or a future vendor kernel, but it should not be assumed to accelerate the RV2 CPU path.

The scratch CTC experiment

The scratch model was intentionally small and CTC-based. CTC is a clean speed proof because the encoder dominates, greedy decode is nearly free, and there is no autoregressive token loop or transducer joiner loop. If a tiny CTC encoder could not get large realtime headroom, a more complex architecture probably would not rescue the speed target.

The later scratch branch used a 384-wide, 10-layer encoder with a 512-token SentencePiece CTC target and a small token bigram language model. On full LibriSpeech test-clean, the best decoded setting reached about 0.321 WER and 0.138 CER. That is not competitive with the shortlist models. It is, however, fast.

The full-pipeline result includes feature extraction, fixed 800-frame ONNX chunks, and decoding. Greedy CTC lands inside the original 0.15-0.20 RTF target band. Beam search with a token bigram LM is still about 4.6x faster than realtime, but the Python decoder pushes the median to about 0.217 RTF.

What that means

The scratch CTC path proves that RV2-class hardware has enough CPU for very fast local STT if the model is designed for it. It does not prove that the scratch model is useful yet. Accuracy is the blocker. A custom CTC path probably needs a better training recipe, stronger data, teacher distillation, or a more careful decoder before it beats practical off-the-shelf candidates.

The speed margin is real but smaller than the encoder-only benchmark suggested. Encoder/logit timing alone made the scratch CTC branch look roughly 4-7x ahead of the shortlist. Once real features and beam+LM decoding were included, the practical advantage narrowed to something closer to 1.7-3.1x versus the better loaded-model candidates.

Practical conclusion

If the goal is useful local STT soon, start with Moonshine tiny, Parakeet realtime EOU, Parakeet TDT int8, or a sherpa Paraformer-style control path. Keep Vosk grammar mode for constrained command interfaces. Treat Qwen-style and Granite-style models as accuracy or architecture references on this board, not realtime defaults.

If the goal is research, the scratch CTC path is worth continuing. It gives a clear speed ceiling, a simple export path, and a narrow optimization target: int8 encoder compute plus a cheaper decoder. The next high-value experiment is not making it smaller. It is making it accurate enough that the speed advantage matters.
