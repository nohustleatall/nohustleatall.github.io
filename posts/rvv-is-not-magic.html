RVV notes

RVV is not magic

The RISC-V Vector Extension is real and useful on the Orange Pi RV2. It also does not make the whole board fast. The latest measurements point to a narrower rule: RVV pays when the kernel is regular, explicit, reuse-heavy, and exported into a runtime path that actually uses it.

The misconception

It is easy to talk about RVV as if it were a general acceleration switch. That is not what the measurements show. Text parsing, formatting, pointer-heavy work, filesystem traversal, scalar DSP, and ordinary Linux utility tasks still tend to favor the Pi 400 by a wide margin. RVV does not rescue those workloads just by existing.

Where RVV becomes interesting is more specific: explicit int8 kernels, dense quantized inference, some MiniLM embedding shapes, and linear binary stream work where the CPU can keep moving predictably through memory.

Auto-vectorization was not enough

The first int8 dot benchmark separated three things: scalar performance, compiler auto-vectorization, and explicit vector intrinsics. On this loop, the RV2 compiler did not discover the useful RVV path. The Pi 400 compiler did find a useful NEON path.

K RV2 scalar RV2 auto RVV intrinsic NEON intrinsic
256 0.197 GMAC/s 0.189 GMAC/s 2.313 GMAC/s 4.949 GMAC/s
1,024 0.198 GMAC/s 0.197 GMAC/s 2.667 GMAC/s 5.096 GMAC/s
4,096 0.197 GMAC/s 0.197 GMAC/s 2.786 GMAC/s 5.328 GMAC/s

Explicit RVV turned an unusably slow scalar path into a real vector kernel: roughly 12-14x faster than RV2 scalar. But the explicit NEON kernel on Pi 400 was still about 1.9-2.1x faster than the simple RVV dot kernel. That is the right level of claim: RVV helps a lot versus RV2 scalar, but it does not automatically beat a mature ARM/NEON baseline.

Reuse helps, but does not change the story

A plain dot kernel is pessimistic for matrix multiply because it pays a horizontal reduction cost for every output. A more realistic kernel should reuse the loaded activation vector across multiple output columns. The follow-up 1x4 tile test checked exactly that.

Shape RVV dot RVV 1x4 RVV gain NEON 1x4 NEON/RVV
1x4x256 2.224 2.630 1.18x 5.114 1.94x
4x4x256 2.260 2.799 1.24x 5.307 1.90x
1x4x1024 2.634 2.951 1.12x 5.623 1.91x
4x4x1024 2.659 2.991 1.12x 5.693 1.90x

Reuse matters. The RVV 1x4 kernel improved by about 12-24% on actual multi-output shapes. But that modest reuse did not close the NEON gap in this simple implementation. The dot benchmark was somewhat pessimistic, not fundamentally misleading.

Int4 was not a free win

The low-bit GEMM experiments showed another useful boundary. Int4 values expanded to int8 compute at roughly int8 speed, but packed int4 in the hot loop lost because unpack and sign-adjust overhead dominated. A direct packed-nibble RVV dot was correct, but still much slower than int8 RVV on the tested ASR-like shapes.

For this board, int8 is the first serious optimization target. Packed int4 can still matter for storage or load-time compression, but it should not be assumed to accelerate compute unless the kernel avoids paying unpack cost repeatedly.

Where RVV-style work did help

The positive cases have a pattern.

  • Explicit RVV int8 dot made RV2 much faster than its scalar path.
  • MiniLM int8 embeddings stayed much closer to Pi 400 than general CPU utilities did.
  • The local-search pipeline stayed within roughly 20-25% of Pi 400 because embedding dominated the runtime.
  • A framed binary stream gateway tied Pi 400 because the pipeline was linear, fixed-format, and mostly validation/copy/checksum.
  • Some quantized ONNX dense shapes landed well enough that RV2 could narrow the gap or even slightly win a specific QLinearMatMul shape.

Those are not random wins. They are workloads with regular memory access, predictable loops, useful batching or reuse, and less time spent in branchy text or formatting code.

Where RVV did not rescue the board

The negative cases are just as important. Pi 400 stayed well ahead on decimal parsing, ASCII classification, function-call-heavy loops, allocation churn, byte hashing, source search, JSON parsing, diff generation, filesystem-ish tasks, and scalar audio front-end code that was not explicitly vectorized.

That is why "RVV exists" is not a sufficient performance plan. A workload can be CPU-bound and still be a poor RVV target.

Practical checklist

Use RVV when the kernel has most of these properties:

  • regular contiguous loads and stores;
  • enough K or loop length to amortize setup;
  • multiple outputs, batching, or other data reuse;
  • predictable control flow;
  • checksum-verifiable output;
  • an implementation path that actually emits RVV instructions.

Be skeptical when the workload is mostly text parsing, formatting, pointer chasing, small unpredictable branches, filesystem traversal, or packed low-bit compute that must unpack inside the hot loop.

Conclusion

RVV is worth using on the RV2, but it is not magic and it is not automatic. The current compiler stack did not find the useful int8 vector path in plain C. Explicit intrinsics worked and helped a lot, but a simple NEON baseline on Pi 400 was still faster for the same int8 primitive.

The most productive way to use this board is to treat RVV as a kernel-design tool, not a marketing feature. Pick regular, batched, reuse-heavy work; prove scalar correctness first; write or choose a runtime path that actually uses RVV; then measure the full pipeline.
