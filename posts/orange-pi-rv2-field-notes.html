Orange Pi RV2 Field Notes

A practical checklist for using the Orange Pi RV2 as a RISC-V Linux development machine, especially if you want to run coding agents or experiment with local AI runtimes.

Known-good baseline

The notes here come from an Orange Pi RV2 / SpacemiT K1 class board running a vendor Linux 6.6.x image. Treat that as the baseline. Other images may differ in kernel config, device nodes, package availability, and RVV behavior.

uname -a
cat /etc/os-release
lscpu

First boot checklist

Install a basic development and sandboxing toolchain before debugging higher-level software.

sudo apt update
sudo apt install -y bubblewrap git curl file build-essential pkg-config cmake ninja-build python3 python3-venv python3-pip
bwrap --version

Codex’s Linux sandbox path expects system bubblewrap on this build. Without it, sandboxed command execution will fail before the interesting work starts.

Install Codex

Download the current RISC-V build, verify the checksum, make it executable, and run the version check.

curl -LO https://github.com/nohustleatall/codex-riscv-builds/releases/download/v0.125.0-orangepi-rv2-riscv64gc-v8/codex-0.125.0-orangepi-rv2-riscv64gc-v8
curl -LO https://github.com/nohustleatall/codex-riscv-builds/releases/download/v0.125.0-orangepi-rv2-riscv64gc-v8/SHA256SUMS
sha256sum -c SHA256SUMS
chmod +x codex-0.125.0-orangepi-rv2-riscv64gc-v8
./codex-0.125.0-orangepi-rv2-riscv64gc-v8 --version

Expected version:

codex-cli 0.125.0

Hardware reality check

The local investigation did not find an obvious discrete NPU. Device tree, kernel config, device nodes, and runtime behavior all pointed toward CPU/RVV-based INT8 acceleration rather than offload to a separate neural processor.

The practical takeaway: use the board as a RISC-V CPU and RVV development target. Be cautious with marketing-style TOPS claims unless you can map them to a specific exposed device, driver, runtime, and workload.

RVV status

RVV is real and useful on this board. In local microbenchmarks, effective vector length for 8-bit elements was 32 lanes, consistent with a 256-bit vector length. vsetvli measured around 3.4 to 3.7 cycles per call.

lscpu | grep -i vector
cat /proc/cpuinfo | grep -i '^isa' | head

For kernel work, the biggest lesson was simple: RVV helps most when the kernel reuses data and keeps vector accumulators live. Streaming INT8 dot/MAC loops improved modestly. Reuse-heavy micro-GEMM improved much more.

AI runtime notes

The vendor ky-ort package appears to be a CPU/RVV-oriented ONNX Runtime stack with a SpaceMIT/Ky execution provider. Static inspection showed quantized kernels and fused MLP paths, not a clear NPU offload path.

A small INT4 Qwen2 demo produced roughly 13 tokens per second in quick testing. Relaxing permissions for scratch/cache-related device nodes removed warnings but did not materially change throughput. Some profiling and provider-option paths crashed, so treat this stack as useful but fragile.

Performance guidance

  • Hoist vsetvli out of inner loops when possible.
  • Prefer reuse-heavy kernels over simple streaming loops when trying to benefit from RVV.
  • For measured INT8 micro-GEMM, M=4, N=32, and K at least 256 was the best region.
  • For ragged N, padding to 32 and masking stores beat reduced-vector-length tails in the measured cases.
  • Do not treat per-core RVV results as equivalent to board-level marketing TOPS claims.
