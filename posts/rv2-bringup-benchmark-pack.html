Bring-up tools

A first-run benchmark pack for Orange Pi RV2

If you are bringing up a new RV2-like board, the fastest useful test is not a huge model. It is a compact diagnostic that tells you whether scalar CPU work, RVV, ONNX Runtime, and appliance-style pipelines are healthy enough to trust larger experiments.

Why this exists

Most RV2 failures are easy to misclassify. A slow model may be a runtime operator problem. A bad vector result may be compiler auto-vectorization, not hardware. A promising stream workload may only work for fixed binary frames, not text transformation. The bring-up pack is meant to answer those questions early.

Run it

From the lab tree on the board:

python3 scripts/bringup-pack.py --quick --out-dir bringup-pack-out

For a fuller run:

python3 scripts/bringup-pack.py --out-dir bringup-pack-out

If ONNX or ONNX Runtime is not installed yet:

python3 scripts/bringup-pack.py --quick --skip-ort --out-dir bringup-pack-out

The main outputs are bringup-report.md, manifest.json, and one JSON file per benchmark.

What it checks

Area Probe Question
Scalar CPU Scalar, branch, pointer chase, memory, fp32/int8 dot Is ordinary core work healthy or obviously weak?
Native primitives Parsing, classification, calls, allocation, syscalls, qsort, hashing, atomics Will ordinary Linux programs feel reasonable?
Vector path Scalar, auto-vectorized, and explicit RVV/NEON int8 dot Does the compiler find vector code, and do explicit vectors work?
Reuse threshold Dot vs 1x4 matmul tile Does output-column reuse improve the vector kernel?
Appliance shape Framed stream gateway, redaction, audio frontend, protocol bridge Is this board plausible for fixed binary stream work?
Optional ONNX fp32 dense, int8 matmul, QLinearMatMul, quantized conv forms Can the runtime load and run key graph families?

How to read it

If scalar, branch, pointer-chase, and text parsing results are weak, the board is probably not a good general Linux CPU target. That does not make it useless, but it changes what you should try first.

If auto-vectorized int8 dot stays near scalar while explicit RVV is much faster, the compiler is not enough. Hot loops need intrinsics, assembly, or a runtime/library that already maps the shape well.

If ONNX quantized operators fail to load, model benchmarks are not trustworthy yet. Fix the runtime lane or export format before judging model quality or speed.

If the framed gateway is strong but the text and JSON-ish pipelines are weak, the board is better suited to fixed binary stream work than text-heavy transformation.

What this does not replace

The pack is a first-run diagnostic, not a full CPU validation lab. It will not tell you whether a production service is stable for weeks, whether a model is accurate enough, or whether a specific external device path is reliable. It gives you a fast map of where to spend the next day.

Practical use

Run the pack before porting a large model, writing a custom kernel, or treating a board as a deployment target. Save the report with your experiment notes. If a later result looks strange, the bring-up pack gives you a baseline for whether the board, compiler, vector path, and runtime were healthy at the start.

No analytics, comments, or tracking scripts are used here.
