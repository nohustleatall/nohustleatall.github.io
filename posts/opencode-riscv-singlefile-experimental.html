opencode now has an experimental RISC-V single-file build

The earlier Bun/opencode experiments were mostly about finding the runtime wall. The current result is more useful: an experimental opencode 1.14.25 package that runs on an Orange Pi RV2 class RISC-V Linux board.

What exists

The current artifact is a release-shaped tarball named opencode-1.14.25-linux-riscv64-singlefile-experimental.tar.gz. It contains a RISC-V opencode executable, runtime sidecars, native PTY support, a RISC-V Parcel watcher package, package metadata, checksums, and bundled license notices.

sha256: 2ac79866db10c6e81a6df7904c4457b10c61015b4253d4114352b8153f07b29b
target: linux-riscv64-gnu
board class: Orange Pi RV2 / SpacemiT K1 / Ky X1
opencode: 1.14.25
reported build version: 1.14.25-riscv-probe

The package is intentionally labeled experimental. It is not the stock upstream opencode release process with a new target added. It is a RISC-V-focused package built through a local Bun single-file path and a set of scoped compatibility shims.

What passed on RV2

The extracted package passed a remote package smoke test on the RV2. The smoke verified the tarball checksum, extracted the package, verified the package SHA256SUMS, and ran startup and server checks from the extracted directory.

Area Verified surface
Startup --version, --help, debug paths
Data commands DB query, session list, stats, import, export
Models/providers Provider listing, model snapshots, fake-SSE provider streaming path
Server Reduced serve --once, plugin visibility, MCP route visibility

TUI Focused OpenTUI/Solid attach path starts, focuses input, submits quit, and exits cleanly

How the package is shaped

The package is not just one executable. The single-file binary still needs runtime sidecars on this board class, so the package includes rv2-runtime-libs and expects LD_LIBRARY_PATH to point there before startup.

cd opencode-1.14.25-linux-riscv64-singlefile-experimental
export LD_LIBRARY_PATH="$PWD/rv2-runtime-libs${LD_LIBRARY_PATH:+:$LD_LIBRARY_PATH}"
sha256sum -c SHA256SUMS
./opencode --version
./opencode --help

The package metadata records the executable, runtime libraries, librust_pty.so, and the RISC-V Parcel watcher native module. License coverage is packaged under LICENSES/, with a short THIRD_PARTY_NOTICES.md describing the bundled components.

What is still experimental

The biggest qualification is that some surfaces are deliberately reduced or shimmed for the single-file RISC-V package.

  • attach uses a focused OpenTUI/Solid attach path rather than the full upstream TUI app graph.
  • The OpenTUI native library is extracted to a real temporary filesystem path before renderer startup because it cannot be opened directly from Bun's embedded filesystem.
  • bisect-server is a reduced diagnostic command covering the currently useful RISC-V boundaries.
  • The plugin service supports configured local file plugins and safe built-in auth/provider hooks, but skips some optional auth plugins, npm plugin installation, and the server SDK client path.
  • The MCP service is usable for local stdio servers and tool discovery, but the extra /mcp/tools, /mcp/prompts, and /mcp/resources HTTP endpoints are diagnostic surfaces rather than upstream OpenAPI routes.

Why this matters

This changes the status from "Bun-based agents hit the runtime wall" to "a useful experimental opencode package can be made to run on RV2." That is a significant difference for people trying to use RISC-V boards as development machines. It also narrows the remaining work: the next problems are not generic "RISC-V cannot run this" problems, but specific standalone compiler, optional dependency graph, plugin service, and full TUI graph boundaries.

The practical lesson is the same as the Codex build: RISC-V support usually arrives as a pile of small, explicit boundaries. Once those boundaries are named, they become testable.
